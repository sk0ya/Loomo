## 6. 主要コンポーネント設計

### 6.1 サービス抽象（Abstractions）

中核の3サービス。抽象は `src/Loomo.Core/Abstractions/` でインターフェースごとのファイルに分割する。

```csharp
public interface ITerminalService
{
    Task<CommandResult> RunCommandAsync(string command, CancellationToken ct);
    void SetWorkingDirectory(string path);
    bool TryRunInVisibleTerminal(string command);             // 可視ターミナルへ送る（LSP導入等）
    string CurrentDirectory { get; }
    bool IsExecuting { get; }
    event EventHandler<CommandResult>? CommandExecuted;
}

public interface IEditorService
{
    Task OpenFileAsync(string path, CancellationToken ct = default);
    Task<string> GetActiveContentAsync(CancellationToken ct = default);
    Task<string> GetSelectedTextAsync(CancellationToken ct = default);
    Task OpenDocumentAsync(EditorDocument document, CancellationToken ct = default);
    string? ActiveFilePath { get; }
}

public interface IWorkspaceService
{
    string? RootPath { get; }
    string? SelectedPath { get; set; }       // FolderTreeの選択
    void OpenFolder(string rootPath);
    Task<IReadOnlyList<FileNode>> ListAsync(string path, CancellationToken ct = default);
    Task<string> ReadFileAsync(string path, CancellationToken ct = default);
    string ResolvePath(string path);         // ルート配下制限込みのパス解決
    event EventHandler<string?>? SelectionChanged;
    event EventHandler<string?>? RootChanged;
}
```

> 旧 `IEditorService.ShowDiffAsync`/`ApplyEditAsync` は `propose_edit` 撤去（§25）に伴い削除。差分は
> `edit_file`/`write_file` の承認カード（`DescribeInvocation`）で見せる。`Abstractions/` には他に
> `IApprovalService.cs`・`IBrowserService.cs`（`web_search`／可視ブラウザ操作）・`IWorkspaceSearchService.cs`
> （ripgrep ベースのファイル名/全文検索・コマンドパレットと Search パネルの共有バックエンド）・
> `ITestDiscoveryService.cs` も置く。各抽象を機能単位で独立させ、総称的な定義ファイルへ戻さない。

これらの **実装** は `Loomo.Services` に置き、sk0ya.Terminal / sk0ya.Editor のコントロールをラップする。
（sk0ya.* の実APIに合わせてアダプタ内で吸収。API差異の影響をこの層に閉じ込める。）

### 6.2 AIクライアント

```csharp
public interface IAiClient
{
    IAsyncEnumerable<AgentEvent> StreamAsync(
        Conversation conversation,
        IReadOnlyList<ToolDefinition> tools,
        CancellationToken ct);
}
// AgentEvent = TextDelta | ThinkingDelta | ToolUseRequested | TurnCompleted ...
```

ストリーミングで `TextDelta`（逐次テキスト）・`ThinkingDelta`（思考）・`ToolUseRequested`（ツール呼び出し要求）を流す。

**実装は `OnnxGenAiClient` のみ**（プロバイダ `Local`）。**HTTP も外部サーバも無い in-process 推論**で、
`ILocalInferenceEngine` を呼ぶ。エンジンは `LocalInferenceRouter` が `modelPath` で振り分ける：`.gguf`
ファイル → **llama.cpp**（`LlamaCppEngine`／LLamaSharp）／`genai_config.json` を持つフォルダ → **ONNX
Runtime GenAI**（`OnnxGenAiEngine`、CPU int4）。エンジンはモデルを常駐させ（ロードは一度きりの cold cost）、
KV プレフィックスキャッシュが効く。出力は `TextDelta` と最終 `AiUsageReported`（トークン数＋load/prefill/decode の
自己計測 `Stopwatch` 値）として流す。

プロンプトはモデルの `ChatFormat` で分岐（`ChatPrompt` ディスパッチャ）：Phi-4 は `Phi4PromptFormatter`、
Qwen3 は `Qwen3PromptFormatter`（ChatML・no-think）。ツール呼び出しは Phi-4 が本文の JSON 配列、Qwen3 が
Hermes `<tool_call>` ブロックで、`ToolCallTextParser` が `ToolUseRequested` に変換する。モデル別の最適化は
`ModelProfiles`（コンテキスト窓 `NumCtx`・サンプリング・`ChatFormat` をモデル名から解決）が担う。
クラウドAPI（Anthropic / OpenAI / Copilot）にも外部 Ollama にも接続しない。詳細は §25。

## 7. AI連携設計（最重要）

### 7.1 ツール（Tool）= AIの手足

AIエージェントは各プロバイダの function calling / tool use 形式を通じて以下のツールを呼び出す。各ツールはサービスを叩くだけの薄いラッパ。

> **現行は 4 ツール構成**（§25 で再編）。読み取り・検索・一覧・ビルド・テストはすべて `run_powershell` で
> PowerShell として表現し、ファイル書込だけ構造化ツールに分けている（小型ローカルLLMが苦手な PowerShell×JSON
> の二重エスケープを回避するため）。旧 6 ツール（`list_directory`/`read_file`/`open_in_editor`/`propose_edit`/
> `get_selection` ＋ `run_command`）は廃止。

| ツール名 | 接続先サービス | 説明 | 承認 |
|----------|----------------|------|------|
| `run_powershell` | ITerminalService | PowerShell コマンド行を実行し stdout＋exit code を返す（読み取り・検索・一覧・ビルド・テストの汎用ツール） | **必要** |
| `write_file{path,content}` | IWorkspaceService/IEditorService | ファイルを作成/上書きしエディタで開く | **必要** |
| `edit_file{path,old_string,new_string}` | IWorkspaceService/IEditorService | 一意の完全一致で置換（0/複数一致はクリーンなエラー） | **必要** |
| `web_search{query}` | IBrowserService | 可視ブラウザペインで Bing を引き、回答ボックス＋上位結果を構造抽出 | **必要** |

`write_file`/`edit_file` のパスは `IWorkspaceService.ResolvePath`（ルート配下に制限）で解決する。

```csharp
public interface IAgentTool
{
    string Name { get; }
    ToolDefinition Definition { get; }              // JSON Schema（LLMへ渡す）
    bool RequiresApproval { get; }
    string DescribeInvocation(JsonElement arguments); // 承認カードの要約（file 系は差分プレビュー）
    Task<ToolResult> ExecuteAsync(JsonElement args, CancellationToken ct);
}
```

新しい能力の追加 = `IAgentTool` を実装して `App.xaml.cs` で DI 登録するだけ（`ToolRegistry` が自動収集する **拡張ポイント**）。
ただし小型ローカルLLMではツールが増えるほど選択を誤りやすく反復が増えるため、集合は小さく・互いに重ならない・
動詞名で保つ方針（理由は §25）。

### 7.2 エージェントループ（AgentOrchestrator）

```
ユーザー入力（下部AIバー）
        │
        ▼
[1] Conversation に user メッセージ追加
        │
        ▼
[2] IAiClient.StreamAsync(会話, ツール定義)
        │
        ├── TextDelta ───────────► AIバーに逐次表示
        │
        └── ToolUseRequested
                │
                ▼
        [3] SafetyPolicy.Evaluate
                ├─ Blocked → エラー結果をAIへ返す
                └─ Allowed
                ▼
        [4] RequiresApproval && !AutoApprove?
                ├─ Yes → ApprovalService → AIバーに承認カード表示 → ユーザー判断
                └─ No  → そのまま実行
                │
                ▼
        [5] ToolRegistry.Get(name).ExecuteAsync(args)
                │  （= Terminal実行 / Editor操作 / FS読取）
                ▼
        [6] ToolResult を Conversation に追加 → [2] へ戻る
        │
        ▼
最終テキストで完了（TurnCompleted）
```

- ループ・状態は `Loomo.Core` 側に持たせ、UIはイベント購読のみ。
- 中断（キャンセル）対応：`CancellationToken` を全層に伝播。

### 7.3 シーケンス例：「このプロジェクトをビルドしてエラーがあれば直して」

```
User → AIバー：「ビルドしてエラーを直して」
Agent → run_powershell("dotnet build")           [承認] → 独立プロセスで実行
        → 出力（エラー2件）→ ToolResult
Agent → run_powershell("Get-Content Foo.cs")     [承認] → 内容取得
Agent → edit_file("Foo.cs", old, new)            [承認] → 置換しエディタで開く
User  → 承認
Agent → run_powershell("dotnet build")           [承認] → 再ビルド成功
Agent → 「修正完了。ビルドが通りました」          → AIバーに表示
```

人間が同じ作業をする時の経路（FolderTree→Terminal→Editor）と完全に一致しているのがポイント。
（承認は `tool.RequiresApproval && !AutoApprove` のときのみ。AutoApprove ではカードを省略する。）

## 10. セキュリティ / 安全設計

- **コマンド実行とファイル書込は既定で承認制**（`ApprovalService`）。AIバーの承認カードで内容を確認してから実行。
- 「自動承認」モード（信頼ワークスペース向け）は設定でオプトイン。危険コマンドのブロックリスト/許可リストを用意。
- 実行スコープはワークスペースルート配下に限定（パストラバーサル防止）。
- APIキーは平文保存しない（現状は DPAPI）。
- 会話は `ConversationStore` でセッション保存。詳細なツール実行トレースは §20 の観測性設計で段階導入する。

## 25. in-process 推論（ONNX Runtime GenAI ＋ llama.cpp）＋ ツール再編（2026-06 実装完了）

> **これが AIプロバイダ・ツール構成に関する現行の正本。** §22（外部 Ollama サーバ）を全面置換し、
> §2/§3/§6.2/§7/§12/§13/§14/§20.7 の AI 周りはここに読み替える。「ローカルLLM専用・クラウド非接続」
> という大方針は §22 から不変で、**接続先が外部サーバ → 同一プロセス内**に変わった。

### 25.1 なぜ in-process へ
- 外部 Ollama サーバ依存（別プロセスの起動・死活確認・HTTP）を排し、**1 プロセスで完結**させる。
  配布・オフライン動作・KV キャッシュ制御が単純になる。
- ONNX Runtime GenAI（`Microsoft.ML.OnnxRuntimeGenAI`・CPU int4）で in-process 推論。HTTP は無い。

### 25.2 推論エンジン（`Loomo.Ai/Clients/`）
- **`IAiClient` 実装は `OnnxGenAiClient` のみ**（プロバイダ `Local`）。`AiClientFactory.ResolveCurrent()` が
  毎ターン `AiSettings` を読んで生成するため設定変更は即時反映。
- 実エンジンは `ILocalInferenceEngine`。**`LocalInferenceRouter` が `modelPath` で振り分ける**：
  - `.gguf` ファイル → **`LlamaCppEngine`**（llama.cpp / LLamaSharp / GGUF）
  - `genai_config.json` を持つフォルダ → **`OnnxGenAiEngine`**（ONNX Runtime GenAI / CPU int4）
- `OnnxGenAiEngine`（DI シングルトン・`IDisposable`）が ORT-GenAI の `Model`/`Tokenizer` を所有し、モデルを
  **常駐**させる（ロードは一度きりの cold cost＝CPU で数十秒）。生成は batch-size-1・`SemaphoreSlim` で直列化。
  デコードループ（`AppendTokens`→`GenerateNextToken`→`TokenizerStream.Decode`）はバックグラウンドスレッドで回し、
  `TextDelta` と最終 `AiUsageReported` を `Channel<AgentEvent>` に書く。
- **バックエンド比較（Qwen3-4B・CPU・2026-06）**: decode tok/s は ONNX int4 ≒ GGUF Q4_K_M でほぼ互角、
  モデルロードは GGUF が圧倒的に速い。詳細は `docs/エージェントループ知見.md` §2.2 /
  `docs/reports/llamacpp-vs-onnx-qwen3-4b.md`。
- **ハイブリッド SSM モデルの KV キャッシュ再利用（2026-07-07）**: `LlamaCppEngine` の複数ターン高速化は
  「system+tools の安定プレフィックスを毎ターン再計算せず、位置ベースの部分巻き戻し
  （`MemorySequenceRemove`）で差分だけ処理する」方式だったが、**Attention＋SSM（Gated Delta Net 等）が
  混在するハイブリッドアーキテクチャ**（例：Qwen3.5）では SSM 側の再帰状態が位置ベースの巻き戻しでは
  正しく戻せず、2 ターン目以降の decode が即死するバグがあった。GGUF メタデータの `*.ssm.*` キーで
  ハイブリッド構成を検出し、該当モデルではプレフィックス境界を `GetState`/`LoadState` で
  スナップショット・復元する専用経路へ切替える（境界が不一致なら実測して自己修復）。純 Attention モデルは
  従来の位置ベース部分巻き戻しのまま。

### 25.3 プロンプト形式とツール呼び出し
- モデルの `ChatFormat` で `ChatPrompt` ディスパッチャが分岐（実ターンとウォームアップ＝`LocalLlmWarmupService`
  が同じ経路を通り、ウォーム済み KV プレフィックスが初ターンのバイト前方一致になるようにする）。
  - **Phi-4**：`Phi4PromptFormatter`。`<|system|>…<|tool|>[toolsJSON]<|/tool|>…<|assistant|>`。
    ツール呼び出しは本文の JSON 配列 `[{"name":…,"arguments":{…}}]`。
  - **Qwen3**：`Qwen3PromptFormatter`。ChatML（`<|im_start|>role\n…<|im_end|>`）、ツールは Hermes
    `<tools>…</tools>`、ツール結果は `<tool_response>`、末尾を `<think>\n\n</think>` で **no-think** に固定。
- どちらも `OnnxGenAiClient` が全文をバッファし、`ToolCallTextParser.Parse`（共有）が `<think>` 除去・
  `<tool_call>` 抽出・関数呼び出し形式/素の引数/別名キー/コードフェンス対応を行い `ToolUseRequested` に変換。
- システムプロンプトは英語指示／日本語出力。`ChatFormat` で 2 種：`DefaultSystemPrompt`（Phi-4・JSON 配列例）と
  `Qwen3SystemPrompt`（Hermes 例＋no-think）。

### 25.4 モデルプロファイルと取得
- `ModelProfiles.Resolve(model)` がモデル/フォルダ名から `ModelProfile`（コンテキスト窓 `NumCtx`・サンプリング・
  `ChatFormat`）を返す。実効コンテキスト窓（`ProviderConfig.NumCtx` 上書き → プロファイル）はエンジンの
  `max_length` と履歴トリム予算で共有する。
- `ModelDownloadService.Catalog` がダウンロード可能な ONNX（CPU int4・ORT-GenAI 互換）モデルの正本。
  `%APPDATA%/Loomo/models/<FolderName>/` へストリーム保存（再開・キャンセル可）。`ModelCatalogService` が
  そのルート配下のローカルモデルフォルダ（`genai_config.json` 持ち）／GGUF を列挙（HTTP 無し）。
  ※ 具体的なモデル名はドリフトするためここに固定しない（出所＝`ModelDownloadService.Catalog`）。

### 25.5 ツール再編（6 → 4）
- 旧 6 ツール（`list_directory`/`read_file`/`open_in_editor`/`propose_edit`/`get_selection`＋`run_command`）を撤去し、
  **`run_powershell`・`write_file`・`edit_file`・`web_search`** の 4 つに集約（§7.1）。
  - **`run_powershell`**（`TerminalTools`/`PwshTool`）が読み取り・検索・一覧・ビルド・テストの汎用ツール。
    実行は可視ターミナルに流さず独立プロセス（§13）。
  - **`write_file{path,content}`** / **`edit_file{path,old_string,new_string}`** はファイル書込を JSON 引数として
    渡し、小型 LLM が苦手な PowerShell×JSON の二重エスケープを回避。パスは `ResolvePath` でルート配下に制限。
  - **`web_search{query}`** は可視ブラウザペインで Bing を引き、回答ボックス＋上位結果を構造抽出。
- **なぜ少数か**：旧「ツール数 ∝ prefill」は Ollama 時代の測定で、ONNX では `system+tools` の安定プレフィックスが
  KV キャッシュ再利用される（§25.2）。残るコストは**選択信頼性**（小型 LLM は選択肢が増えると誤りやすい）なので、
  集合を小さく・互いに重ならない・動詞名で保つ。4 ツールとも `RequiresApproval`（AutoApprove 時のみカード省略）。

### 25.6 usage テレメトリ
- `OnnxGenAiEngine.RunSync` が `Stopwatch` で load/prefill/decode を**自己計測**し、トークン数とともに
  `AiUsageReported`（→ `ai.usage` トレース・§20）として流す。AI バーの「📊 AI内訳」（`AiBarViewModel.FormatUsage`）に
  ライブ表示。ローカル推論のため金銭コストは無い。
