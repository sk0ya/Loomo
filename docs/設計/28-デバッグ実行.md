# §28 デバッグ実行（IDEペイン）

> 作成日: 2026-07-07
> 位置づけ: `PaneKind.Debug`（§26.1 表では一行のみの記載だった）の**正本**。DAP（Debug Adapter Protocol）を
> 使ったビルド・テスト・デバッグ実行機能。実装は `src/Loomo.Core/Debug/`（モデル・純ロジック）・
> `src/Loomo.Services/Debug/`（DAPクライアント）・`src/Loomo.App/ViewModels/Debug*.cs`・
> `src/Loomo.App/Views/{DebugView,DebugConfigView,DebugVariablesView,DebugBreakpointsView,
> DebugImmediateView}.xaml(.cs)` ＋ `ShellWindow.Debug.cs`（エディタ⇔デバッグの橋渡し）。

---

## §28.1 目的

「ビルド・テスト・デバッグ」という IDE的な操作を、Loomo のペインの1つとして持ち込む。VS Code 的な
DAP（Debug Adapter Protocol）連携により、Editor（`sk0ya.Editor.Controls`）のブレークポイントガター・
実行行ハイライト・DataTip（ホバー評価）と噛み合わせる。対象は **C#/.NET（`netcoredbg` アダプタ）のみ**。

---

## §28.2 アーキテクチャ（ファサード＋機能別サブVM）

`DebugViewModel`（`IDebugSession` 実装・`IDisposable`）がペインのファサードで、`ShellWindow.xaml` から
`DataContext="{Binding Debug}"` で束ねられる。セッション横断の状態（`StatusMessage`/`IsBusy`/`IsStopped`/
`IsTaskRunning`/`IsAdapterMissing`／コンソール `Output`＝上限2000行）を持ち、`IDebugService` のイベント
（`Output`/`StateChanged`/`Stopped`/`Continued`/`Exited`）を購読してペイン状態へ翻訳し、エディタ連携イベント
（`ExecutionLineChanged`/`StoppedChanged`/`FramePreviewRequested`/`FrameActivated`/`BreakpointsRefreshed`）を発火する。

**サブVMは6個**（機能別に分割、いずれも `internal` コンストラクタで `DebugViewModel` が自ら生成——DIでは
個別解決しない）：

| サブVM | 担当 |
|--------|------|
| `Launch`（`DebugLaunchViewModel`） | 開始/停止/再起動/ステップ・手動ビルド・例外フィルタ・Set Next Statement・Run to Cursor・Step Into Target。主にヘッダーツールバーから叩かれる |
| `Profiles`（`DebugProfilesViewModel`） | 起動構成（プロファイル）一覧・プロジェクト選択・永続化との仲介（§28.4） |
| `Inspection`（`DebugInspectionViewModel`） | 停止中だけ意味を持つもの：コールスタック・変数ツリー・Autos・Watch・スレッド・イミディエイト・モジュール |
| `Breakpoints`（`DebugBreakpointsViewModel`） | ブレークポイントの単一の真実（`Dictionary<path, List<BreakpointViewModel>>`）。`IDebugService.SetBreakpointsAsync` へ反映 |
| `Attach`（`DebugAttachViewModel`） | 実行中プロセスの列挙・フィルタ・アタッチ |
| `Tests`（`DebugTestsViewModel`） | テスト発見・実行（`dotnet test`＋TRX）。DAP セッションとは独立（`ITerminalService` 経由） |

各サブVMは `DebugViewModel` に直接依存せず、共有の「狭い腰」**`IDebugSession`** インターフェース
（`IsBusy`/`IsStopped`/`IsTaskRunning`/`IsAdapterMissing`/`StatusMessage`・`BeginSession()`/`CancelSession()`
（`CancellationTokenSource` をラップ）・`Append`/`WriteConsole`・`RequestOutput()`・`FindBuildTarget()`・
`RaiseExecutionLine`/`RaiseFramePreview`/`RaiseFrameActivated`/`RaiseBreakpointsRefreshed`・
`SessionStateChanged` イベント（各サブVMの `RelayCommand.CanExecute` 再評価用））に依存する。

`ShellWindow.Debug.cs` はエディタ⇔デバッグの橋渡し（View層、ViewModelではない）：ブレークポイントガターの
トグル・実行行ハイライト・フレームプレビュー/ジャンプ・DataTip の有効/無効・ガターグリフのマッピング
（`BreakpointGlyphInfo` → `EditorBreakpoint`）を配線する。

DI: `DebugLaunchProfileStore` と `DebugViewModel` を `App.xaml.cs` で Singleton 登録。`DebugViewModel` は
コンストラクタで `IDebugService`/`IWorkspaceService`/`ITerminalService`/`ITestDiscoveryService`/
`DebugLaunchProfileStore` を受け取り、6サブVMを自前で構築する。

## §28.3 DAP 連携

- **アダプタ**：`DebugAdapterCatalog`（`Services/Debug/`）に登録された唯一のアダプタは **`netcoredbg`**
  （Samsung 製 OSS。実行ファイル `netcoredbg`、引数 `--interpreter=vscode` で stdio DAP モードへ）。
  拡張子 `.cs`。導入コマンドは `scoop install netcoredbg`。**MS 公式 `vsdbg` は VS/VS Code 外の利用がライセンスで
  禁止されているため意図的に不採用**（コードコメントに明記）。解決は LSP サーバカタログと同じ流儀で
  **PATH 検出のみ**（`ExecutableResolver.IsOnPath`）。バージョン固定・同梱バイナリは無い。
- **トランスポート**：`DapProtocolClient`（`internal sealed class`）が低レベル層。子プロセスとして起動
  （stdin/stdout/stderr リダイレクト・UTF-8 no-BOM・シェル/ウィンドウ無し）、フレーミングは **LSP と同じ
  `Content-Length: N\r\n\r\n<json>`**（DAP の JSON 本体形＝`seq`/`type: request|response|event`）を使う旨
  コードコメントに明記。`SendRequestAsync` は `seq` を採番し `TaskCompletionSource<JsonElement?>` を
  `_pending` 辞書に積み、`ReadLoop`（専用バックグラウンド `Thread`）が `response` を `request_seq` で
  マッチさせて完了させる。`event` は `EventReceived` で通知。アダプタからの**逆方向リクエスト**
  （例：`runInTerminal`）は即座に `success:true`／空ボディで応答してブロックを防ぐ。診断用に全送受信を
  `%TEMP%/loomo-dap-debug.log`（1行300文字に切詰め）へ記録する。`Dispose()` はプロセスツリーごと kill。
- **セッション状態機械**：`NetcoredbgDebugService`（`IDebugService` 実装）が実体。起動シーケンス：
  `initialize` → 応答 → `launch`/`attach` を送信（まだ待たない）→ アダプタが `initialized` イベントを
  発火 → `ConfigureAsync()`（記憶済みブレークポイントをソース単位で送信＋例外フィルタ＋`configurationDone`）
  → `configurationDone` 完了後にようやく `launch`/`attach` の応答が返る → `SetState(Running)`。
  `_requestSent` という `TaskCompletionSource` で「`launch`/`attach` を stdin へ書き終えてから
  `configurationDone` を送る」順序を保証する（netcoredbg は `configurationDone` が先に届くと15秒でタイムアウト
  する。`initialize` の await と `initialized` イベントが競合し得るための対策）。
  - `launch` 本体：`{name, type:"coreclr", request:"launch", program, args, cwd, env, stopAtEntry, justMyCode,
    console:"internalConsole"}`。**`console` は常に `internalConsole`**（外部コンソール窓は無い）。
  - `attach` 本体：`{name, type:"coreclr", request:"attach", processId, justMyCode:false}`——**アタッチ時は
    `justMyCode` が常に false に固定**（起動時のチェックボックスは launch にのみ効く）。
  - Capability ネゴシエーション：`initialize` 応答の `supportsSetVariable`/`supportsGotoTargetsRequest`/
    `supportsStepInTargetsRequest`/`exceptionBreakpointFilters` を読み、`IDebugService.SupportsSetVariable`/
    `SupportsSetNextStatement`/`SupportsStepInTargets`/`ExceptionFilters` として公開（機能の出し分けに使う）。
  - `setBreakpoints`/`setExceptionBreakpoints`/`continue`/`next`/`stepIn`/`stepOut`/`pause`/`stackTrace`/
    `scopes`/`variables`/`evaluate`（`context: watch` と `repl` を使い分け）/`threads`/`setVariable`/
    `gotoTargets`/`goto`/`modules`/`stepInTargets` は型付き DAP SDK を使わず、生の `JsonElement` で手組み。
  - 停止：`disconnect` を送信（`terminateDebuggee = !_attached`＝launch は対象を道連れに終了、attach は
    対象プロセスを生かしたまま切り離す）、2秒タイムアウト、常にクライアントを破棄する。

## §28.4 起動構成（プロファイル）

`3a56aa8`（2026-07-07）で追加。

- **モデル** `DebugLaunchProfile`（`Core/Debug/`）：不変 `record`——`Id, Name, ProjectPath（相対・null=自動検出）,
  TargetProgram, BuildFirst, LaunchArgs, LaunchEnv, JustMyCode, BreakOnAllExceptions, BreakOnUncaughtExceptions`。
- **永続化** `DebugLaunchProfileStore`：`%APPDATA%/Loomo/launchProfiles.json` に **全ワークスペース分を1ファイル**
  （`WorkspaceEntry(RootPath, Profiles, SelectedProfileId)` の配列）で保存。ワークスペースごとに独立した
  プロファイル集合を持つ。壊れたファイルは黙って空扱い。
- **プロジェクト発見** `DebugProjectDiscovery`（静的・UI非依存）：`.sln` があれば正規表現で `Project(...)` 行を
  パースしパス実在チェック、無ければ深さ制限（4）付き再帰スキャン（`bin/obj/.git/node_modules/.vs` 等を除外）。
  `.csproj` は軽量パース（`XDocument`）でテストプロジェクト（`<IsTestProject>true</IsTestProject>` または
  `Microsoft.NET.Test.Sdk` 参照）を判別し起動候補から除外する。先頭には常に **番人エントリ `(自動検出)`**
  （空パス）を置く——WPF `ComboBox.SelectedItem` は `null` を「未選択」（空欄表示）として扱うため、`null` の
  代わりに実体オブジェクトの番人が必要という理由がコメントに明記されている。
- **仲介** `DebugProfilesViewModel`：`IWorkspaceService.RootChanged` で `ReloadForWorkspace()`
  （再発見＋ストア読込。無ければ `"既定"` を1件 seed）。`AttachLaunch(launch)`（`DebugViewModel` 構築直後に
  1回呼ぶ）が `Launch.PropertyChanged` を購読し、選択中プロファイルの値を `Launch` の各バインド可能プロパティへ
  即時反映する。プロファイル切替は `_applying` フラグで書き戻しループを防止。監視対象プロパティ
  （`WatchedLaunchProperties`）の編集や `SelectedProject` 変更は **400ms デバウンス**（`DispatcherTimer`）の後、
  現在値を選択中プロファイルへ `with { ... }` で書き戻して永続化する。
  `SelectedProjectPath` は `(自動検出)` 選択時は `null`、それ以外は選択 `.csproj` の絶対パスとなり、
  `DebugTargetResolver.ResolveProgramAsync` の `explicitProjectPath` へ流れる（`StartAsync` の対象解決にのみ影響。
  ヘッダーの「🔨 ビルド」ボタン＝`BuildTarget` は `FindBuildTarget` を使い続けるため、プロファイルの
  プロジェクト選択には影響されない）。
- **UI**：**ヘッダーの ComboBox**（`ShellWindow.xaml`、幅150・`DisplayMemberPath=Name`）は**切替専用**。
  追加/名前変更/削除・プロジェクト選択は「構成」タブ（`DebugConfigView`）側に限定する
  （XAML コメント「編集…は『構成』タブ側」）。削除は最後の1件では不可（`CanDeleteSelectedProfile`）。

## §28.5 UI 構成（2026-07 再編後）

`DebugConfigView.xaml` 自身のコメントが要約：*「ヘッダ（IDEラベル・状態・デバッグツールバー）はホスト
（ペインヘッダ）側が出す。タブより上には何も置かず、操作はヘッダー、構成（起動構成・例外・アタッチ）は
先頭の『構成』タブに収める。」*

**ペインヘッダー**（`ShellWindow.xaml` の `DebugPane`、左→右）：
1. `IDE` ラベル
2. **起動構成 ComboBox**（§28.4）
3. ステータステキスト（`Debug.StatusMessage`、省略記号つき、残り幅を占有）
4. **デバッグツールバー**（`IsBusy`/`IsStopped` で排他表示の3系統）
   - 非実行時：`▶ 開始` / `🔨 ビルド` / `🧪 テスト` / 「先にビルド」チェックボックス
   - 停止中：`▶` 続行(F5) / `⤼` ステップオーバー / `⤷` ステップイン / `⤴` ステップアウト
   - 実行中：`⏸` 一時停止 / `⟳` 再起動 / `■` 停止
5. ペイン非表示ボタン（`—`）

**ペイン本体**＝`DebugView`（横スクロール対応タブ、フラット下線スタイル）。タブ：構成／出力／変数／自動／
コールスタック／テスト／スレッド／ブレークポイント／イミディエイト／モジュール（`DebugView.xaml.cs` に
`OutputTab=1, VariablesTab=2, TestTab=5` の定数）。自動タブ切替：`IsStopped` になったら **変数タブ**へ、
再開したら **出力タブ**へ。`OutputRequested`（開始/ビルド/テスト/アタッチを押した瞬間に発火。`IsBusy` の
反映より早い）で即座に出力タブへ。テストタブ選択で `Tests.EnsureTestsDiscovered()`。

**「構成」タブ**（`DebugConfigView`）：アダプタ未導入時の導入バー／起動構成セクション（プロファイル
＋/名前変更/削除ボタン・起動プロジェクト ComboBox・対象 `.dll`/`.exe` テキストボックス・起動引数・
環境変数の複数行テキストボックス、**`IsBusy` 中は非表示**）／例外・オプションセクション（「すべての例外」
「未処理の例外」「マイ コードのみ」チェックボックス、常時表示）／アタッチセクション（トグルボタン＋
フィルタ可能なプロセス一覧、**`IsBusy` 中は非表示**）。

**タイトルバートグル**：`DebugPaneToggle`（バーアイコン、ツールチップ「IDE（ビルド・テスト・デバッグ）」）。
`Visibility` はプログラムから制御される（§28.6）。

## §28.6 ワークスペース適用性（C#プロジェクトの無いワークスペースでは非表示）

`59454f9`（2026-07-01）。`DebugTargetResolver.HasCSharpProject(root)` がトップレベル `.sln` 優先、無ければ
深さ制限（3）付き再帰 `.csproj` スキャン（`bin/obj/.git/node_modules/.vs` 等を除外）で判定する
——**無制限 `AllDirectories` を使わない**のは、この判定が初回フレーム描画前に同期実行され、`node_modules` 等
巨大な非C#ワークスペースで起動をブロックしてはならないため。

`ShellWindow._idePaneApplicable`（既定 `true`）＋`ApplyIdePaneApplicability(root)` が `HasCSharpProject` の
結果で `DebugPaneToggle.Visibility` を切替える。**適用外なら以下4箇所すべてから `PaneKind.Debug` を除外**する
（どの経路からも IDE ペインが出てこないように）：
1. タイトルバートグル（上記）。
2. `LoadEnabledSessions` が `_enabledSessions` から `Debug` を除去（保存済みセッション状態からの復活も防ぐ）。
3. `PrepareStageSnapshot`（Stage.cs）が `PaneKind.Debug` を指すソロ/ズーム舞台の復元を拒否し `Editor` へ
   フォールバック。
4. `PaneLayout.cs` の保存済みレイアウト復元が、残存する `Debug` タイルを強制的に `Hidden=true` にする
   （古い保存レイアウトが参照していても）。

判定はワークスペースロード時（`ShellWindow.Workspaces.cs`、`LoadEnabledSessions`/`PrepareStageSnapshot`/
レイアウト復元より**前**）と、ワークスペース無し（`root: null`）分岐の両方で行う。

## §28.7 セッション再利用の安全策（間欠的起動失敗の修正、`9663f79` 2026-07-07）

**バグ1：前セッションのアダプタがファイルハンドルを保持したまま**——デバッグ対象が自然終了すると
`NetcoredbgDebugService.Finish()` が `Exited` を発火するが、**アダプタ本体（netcoredbg）は生き続け**、対象
`.dll`/`.pdb` を開いたままになる。従来テアダウン（`TearDownAdapterAsync`）は fire-and-forget だったため、
直後に「開始」を押すと古いアダプタがDLLをロックしたままリビルドが始まり "ファイル使用中" で失敗し得た。
→ テアダウンタスクを `_teardownTask` に保持し、新設の `IDebugService.WaitForIdleAsync()` がそれを待つ
（既に完了/null なら no-op）。`DebugLaunchViewModel.StartAsync`/`BuildTarget` は対象解決・ビルドの**前**に
これを await するようになった。

**バグ2：明示ターゲット時に「先にビルド」が無視される**——`DebugTargetResolver.ResolveProgramAsync` は
`TargetProgram`（`.dll`/`.exe` パス）が明示されていると、`BuildFirst` が真でもビルドせず `File.Exists` だけ
見ていた。→ `FindProjectNear(programPath)`（対象パスから最大6階層親を遡り `.csproj` を探索）を追加し、
`BuildFirst` が真ならそのプロジェクト（またはプロファイルの明示プロジェクト `explicitProjectPath`）を
`File.Exists` 判定の**前**にビルドするようにした。プロジェクトを推測できない場合はビルドを黙ってスキップ
（従来の存在確認のみへフォールバック）。

## §28.8 実装済み機能（要点）

条件付きブレークポイント／ログポイント（`BreakpointViewModel.Condition`/`HitCondition`/`LogMessage`、
ガターの専用グリフ）・例外ブレーク（全例外／未処理のみ）・変数の書換え（`SupportsSetVariable` かつ
コンテナ参照が非ゼロのときのみ編集可＝スコープ直下ノードは不可）・スレッド切替・一時停止/再起動・
Set Next Statement（`SupportsGotoTargetsRequest` 依存）・マイコードのみ・実行中プロセスへのアタッチ
（`.NET` 判定はロード済みモジュールに `coreclr.dll`/`clr.dll`/`hostpolicy.dll` があるかのベストエフォート）・
コールスタックのプレビュー（フォーカスを奪わないプレビュータブ）/ジャンプ（通常タブ＋フォーカス）・
出力/変数/フレーム等のコピー・Run to Cursor（一時ブレークポイントを `_tempBreakpoints` として持続ブレーク
ポイントとマージ送信し、次の停止で理由を問わずクリア）・イミディエイト（REPL、`evaluate context:"repl"`、
上限500件、副作用がありうるため評価後に変数/ウォッチを再取得）・モジュール一覧・起動引数/環境変数
（`DebugLaunchArgs` が空白区切り＋引用グループ化／`KEY=VALUE` 各行＋`#`コメント無視、Core層の純ロジック）・
Autos「自動変数」（`AutosExtractor`：停止行＋直前行から識別子/メンバーチェーン候補を正規表現抽出し、
コメント/文字列リテラルを除去、C#キーワードを除外——**netcoredbg にネイティブな autos スコープが無いための
ヒューリスティック近似**であることを明記）・Step Into Target（`SupportsStepInTargets` 依存）・DataTip
（停止中のみ有効化）・Watch（停止/フレーム変更ごとに再評価）・ビルド統合（`dotnet build "<target>" -c Debug
--nologo`、対象は `.sln` 優先／なければ最初の `.csproj`）・テスト統合（`dotnet test`＋TRXパース、DAPセッション
とは独立、ファイル監視による再発見）。

## §28.9 既知の制約（意図的に対応していないもの）

- **アダプタは netcoredbg 一本・C#/.NET（`coreclr`）のみ**。他言語/ランタイムは配線されていない。
- **`vsdbg` 非対応**（ライセンス上の理由）。同梱バイナリも無く、PATH 検出のみ・バージョン確認も無い。
- **`console` は常に `internalConsole`**——対象の外部コンソール窓／統合ターミナル起動モードは無い。
- **アタッチ時は `justMyCode` が常に false**（チェックボックスは launch のみに効く）。
- **Autos はヒューリスティック**（netcoredbg ネイティブのスコープではない）。
- **プロセスアタッチの `.NET` 判定はベストエフォート**（権限/ビット不一致で調べられないプロセスは
  非マネージド扱いになる）。
- **ブレークポイントの条件式/ヒット条件/ログメッセージはクライアント側で検証しない**（そのまま netcoredbg へ
  送るだけ）。
- **テスト実行/ビルドは常に別CLIプロセス**（`dotnet test`/`dotnet build`）——DAPセッションを介した
  「テストをデバッグする」（実行中テストへのアタッチ）統合は無い。
- **多重セッション不可**——新規セッション開始は既存セッションを必ず先に停止する（`_client`/`_state` は単一）。
- **起動構成が持つのは `DebugLaunchViewModel` が既に公開しているフィールドのみ**（対象・先にビルド・引数・
  環境変数・マイコードのみ・例外2種・プロジェクトパス）。作業ディレクトリの上書き・複合起動（compound
  launch）・pre/post タスク（VS Code の `tasks.json` 相当）は無い。
