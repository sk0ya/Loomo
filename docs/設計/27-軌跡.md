# §27 軌跡（Thread Rail）

> 作成日: 2026-07-03
> 位置づけ: アイデア.md「Semantic Depth」構想の **Thread Rail の種**。本書は「軌跡」機能の**正本**。
> 実装: `TrailViewModel`（`src/Loomo.App/ViewModels/`）＋ `TrailStore`（`src/Loomo.App/Services/`）＋
> `ShellWindow.Trail.cs`（配線）＋ ShellWindow.xaml Row 2 の全幅バー。

## §27.1 目的と非目的

**目的**：ユーザーが作業中に通過した「地点」（開いたファイル・見たページ・切り替えたペイン／パネル／ターミナルタブ・
変えたレイアウト）を時系列の点（ドット）としてウィンドウ最下部に並べ、**あとから任意の地点へ戻れる**ようにする。
戻るとは「その地点で見えていた画面（対象＋表示コンテキスト）を復元する」こと。記録は日別・ワークスペース別に
無制限で永続化し、過去の日も遡れる。

**非目的（現段階でやらないこと）**：
- 出自（何から何へ遷移したか）の記録・可視化。ドットは点であって辺（遷移の矢印）ではない。
- 分岐・仮説・確度・Evidence など Semantic Depth 構想の上位概念。軌跡は「一次元の通過履歴」に限定する。
- 素材（Material）や操作（Operation）の記録。軌跡はナビゲーション履歴であって作業内容ログではない。
- AI エージェントの操作トレース（それは §20 観測性が担当。軌跡は**人間の**ナビゲーション）。

## §27.2 概念モデル

- **地点（エントリ）**：一度通過した画面の状態のスナップショット。1 ドット＝1 エントリ。
- **種別（`TrailEntryKind`）**：`File` / `Browser` / `Pane` / `Panel` / `Terminal` / `Layout` / `Preview` /
  `Session` / `Edit` / `Git`。値は SQLite へ int 保存するため **末尾追加のみ**。
- **対象（Target）**：戻り先の実体。ファイル＝フルパス、ブラウザ＝URL、ペイン／パネル＝enum 名、
  ターミナル＝永続タブ ID（Guid）、レイアウト＝配置キー、プレビュー＝プレビュー元ファイルのフルパス、
  セッション＝保存済み AI セッションの ID、編集＝編集したファイルのフルパス、Git＝操作種別キー。
  - **Preview（プレビュー地点）**：EditorSupport（プレビュー）ペインは「どのペインか」ではなく
    「**どのファイルを映していたか**」が戻り先の本体なので、Pane 種別ではなく target にプレビュー元
    ファイルパスを持つ独立種別にする。戻ると同じファイルをエディタで開き直し、そのプレビューを開く。
    File と同じく target は大文字小文字を無視してデデュープする（Windows のパス）。
  - **Session（AI セッション地点）**：AI 会話セッションがアクティブになった瞬間（保存済みセッションの**復元**、
    または新規セッションの**初回確定**）を、そのセッション ID を target・タイトルを label にして積む。戻ると
    そのセッションを復元し（`AiBarViewModel.RestoreSessionById`）、AI ペインを前面化してフォーカスする。
    記録は `AiBarViewModel.SessionActivated` イベント経由（UI スレッド発火）で、`RestoreSession` は毎回、
    自動保存（ターン終了）は**新規セッションの初回のみ**発火する（`_lastAnnouncedSessionId` で毎ターンの重複を抑止／
    ジャンプ復帰中は `RecordTrail` の `_trailSuppressed` が抑止）。生存確認は `ConversationStore.Exists`。
  - **Edit（編集地点）**：エディタでファイルを**編集した**瞬間（`VimEditorControl.BufferChanged`）を、target＝
    ファイルのフルパス・line＝編集時のカーソル行にして積む。File（＝ファイルを開いた／見に行った地点）とは
    別種で、「そのファイルを**編集した**」ことを表す。BufferChanged は 1 打鍵ごとに飛ぶので即記録はせず、
    編集が一段落するまで待って（`RecordTrailEdit` の 1.5 秒デバウンス）最新の編集行で 1 点に畳む。未変更
    （読込直後）・無題・仮想ドキュメントは対象外。**内容は復元しない**（要件）——戻ると File と同じく同じ
    file:line を開き直すだけ。デデュープは **行ごと**：パスの大小は無視しつつ、同じ行を編集し続ける間だけ
    畳み、**同じファイルでも別の行を編集したら新しい点を積む**（列は無視。File が path だけで畳むのと違う点）。
  - **Git（Git 操作ログ）**：変更系 git コマンド（コミット・ステージ・プッシュ・ブランチ切替など）を
    **ログとして**積む。target＝操作種別キー（`commit` / `stage` / `push` / `checkout` …）、label＝表示名。
    記録は `GitService.OperationExecuted`（変更系の choke point `MutateAsync` から発火・**成功した操作だけ**記録
    ＝失敗→再試行の二重や多段操作の断片を避ける）を購読し、`DescribeGitOperation` がサブコマンド行を
    種別キー＋ラベルへ変換する（多段の破棄＝clean+restore を同じ `discard` キーで 1 点に畳む）。**ログ専用で
    戻り先を持たない**（要件）——`CanJumpToTrailEntry` で常に false を返し、クリックしても復元しない。配置
    （paneLayout）も載せない。`MutateAsync` はバックグラウンドスレッドなので UI スレッドへ回してから記録する。
- **表示コンテキスト**：対象と一緒に記録する「画面の枠組み」。次の 3 つ。
  - `Mode`（`DisplayMode.Layout` / `Solo`）：タイル表示かソロ（舞台）表示か。
  - `StagePane`：ソロ時に舞台へ立っていたペイン（レイアウト時は null）。
  - `PaneLayout`：そのときのペイン配置 JSON（`PaneNodeSnapshot` を `System.Text.Json` で直列化）。
- **現在地（CurrentIndex）**：軌跡上でいまユーザーがいる地点。ドットを強調表示する。-1 は「現在地なし」。

**不変条件**：あらゆるエントリは「対象」＋「表示コンテキスト」を必ず持つ。戻るときは**表示コンテキストを先に
復元してから対象へ移動する**（§27.6）。順序が逆だと、対象を復元した直後に配置復元が画面を作り替えて破綻する。

## §27.3 記録仕様

### §27.3.1 共通入口

記録は種別を問わず、ViewModel の単一入口 `TrailViewModel.Record(kind, target, label, line, col, mode, stagePane, paneLayout)`
へ集約する。`RecordFile` / `RecordBrowser` / `RecordPane` / `RecordPanel` / `RecordTerminal` / `RecordLayout` は
target/label の作り方が決まっている薄い糖衣で、内部はすべて `Record` に流れる。

View 側は `RecordTrail(record)`（`ShellWindow.Trail.cs`）を唯一の記録経路とし、ここで次を**共通で**行う：
1. 記録抑制中（§27.3.4）なら何もしない。
2. 新しい地点を積む直前に「いま離れるファイル」のカーソル位置を離脱位置へ上書き（§27.3.3）。
3. 現在の `Mode` / `StagePane` / `PaneLayout` を算出し、記録に添える。
4. 最新地点の `PaneLayout` を現在値へ同期（配置だけ変えて対象を変えないケースの取りこぼし防止）。

**新しい軌跡ソースの足し方はこの 3 点だけ**：①`TrailEntryKind` に enum＋`Glyph`（ツールチップ・一意性テスト用）と
`IconGeometry`（バーの絵姿）を対で 1 本ずつ ②`RegisterTrailJumps` に戻る処理を 1 行 ③記録箇所で
`RecordTrail(() => _vm.Trail.Record(...))` を呼ぶ。

### §27.3.2 デデュープ（増殖しない）

`Record` は「直前の最新地点と**同一地点**の再通過」ならドットを増やさず、その行の時刻・ラベル・位置・配置だけを
上書きする。**同一地点の判定**＝ 同一 `kind` かつ 同一 `target` かつ 同一 `Mode` かつ 同一 `StagePane`。
target の一致判定は **File のみ大文字小文字無視**（Windows のパス）、他は完全一致。

ノイズ対策（設計として維持する）：
1. **ペイン切替のデバウンス**：WebView2 実体化やプレビュー更新のフォーカス奪い合いで大量発生するため、
   「同じペインに一定時間（既定 1s）とどまったら 1 個確定」する。行き来（元へすぐ戻る）は保留ごと取り消す。
2. **ファイルを開いているエディタへの切替はファイルのドットが代表**する（ペイン＋ファイルの二重スタック防止）。
3. **直前と同一地点は UPDATE**（上記）。
4. **既定 URL（新規タブの初期表示）と `about:` は記録しない**。
5. **レイアウトの微差は独立ドットにしない**：ペインの**構造**が変わったときだけ Layout ドットを積む。
   リサイズ（Weight だけの変化）やタブ操作でドットが増えてはならない（§27.11-B）。

### §27.3.3 離脱位置の上書き

File 地点の「戻る」は、到着時ではなく**その地点を離れた瞬間**のカーソル位置へ戻す。新しい地点を積む直前
（`RecordTrail` 内）に、最新エントリが現在アクティブなファイルなら、そのタブの現在カーソルで
`line`/`column` を上書きする（`UpdateLatestFilePosition`）。

### §27.3.4 記録抑制

`_trailSuppressed` が真の間は記録しない。立てる場面は 2 つ：
- **ワークスペース切替・起動復元**による機械的なタブ活性化・ナビゲート・レイアウト適用。
- **軌跡からの「戻る」自体**（戻った先を新しい地点として積み直さない）。

> ⚠ 記録抑制と「レイアウト追跡の基準確立」は**別概念**として扱うこと。両者を 1 つのフラグに束ねると、
> 起動直後に File/Browser/Pane/Panel の記録まで巻き添えで止まる（§27.11-A）。

### §27.3.5 記録点（choke point）一覧

| 種別 | 記録トリガ | 呼び出し箇所 |
|------|-----------|-------------|
| File | エディタタブの活性化（無題・仮想ドキュメントは除外） | `RecordTrailEditorTab`（EditorTabs / Tabs） |
| Browser | `NavigationCompleted`（アクティブタブのみ、既定URL/about:除外）／ブラウザペインへのフォーカス確定 | `RecordTrailBrowser`（Browser / PaneFocus 経由） |
| Pane | ペインフォーカス変更（`FocusPane` 末尾＋`GotKeyboardFocus`、デバウンス確定） | `RecordTrailPane`（PaneFocus / Stage） |
| Panel | サイドバーのパネル切替（`ActivePanel` 変更） | `RecordTrailPanel`（xaml.cs） |
| Terminal | ターミナルタブの活性化 | `RecordTrailTerminalTab`（Tabs） |
| Preview | プレビューを明示的に開く（`MarkdownPreviewRequested → OpenEditorSupportAsync`）／プレビューペインへのフォーカス確定 | `RecordTrailPreview`（EditorSupport / PaneFocus 経由） |
| Session | AI セッションのアクティブ化（復元／新規セッションの初回確定） | `RecordTrailSession`（`AiBar.SessionActivated` 購読） |
| Layout | ペイン**構造**の変更（保存の choke point で前回と比較） | `RecordTrailLayoutIfChanged`（Workspaces 保存） |
| Edit | エディタでの編集（`BufferChanged`、未変更/無題/仮想は除外、1.5秒デバウンス確定） | `RecordTrailEdit`（ViewportSplit の `BufferChanged` 配線） |
| Git | 変更系 git コマンドの成功（`MutateAsync` の choke point） | `RecordTrailGit`（`GitService.OperationExecuted` 購読） |

Preview は EditorSupport ペインへの切替を `RecordTrailPane` が横取りして、generic な Pane(EditorSupport) ドット
ではなく「いま映しているファイル」の Preview ドットとして積む（追従先が未確定なら通常の Pane ドットへ落ちる）。
プレビューがアクティブエディタへ**自動追従**して映すファイルを変えた分（＝ユーザーがエディタタブを切り替えた分）は
File ドットが代表するので Preview ドットにはしない（二重を避ける）。Preview ドットは「プレビューを見に行った」瞬間を表す。

Browser も同様に、`RecordTrailPane` がブラウザペインへの切替を横取りして generic な Pane(Browser) ドットではなく
「いま表示しているページ」の Browser ドット（URL／タイトル付き）として積む。ブラウザはナビゲートが起きない限り
`NavigationCompleted` が飛ばないため、既に読み込み済みのページを見に来ただけの「表示」はこの経路がないと軌跡へ
残らなかった。記録すべき URL が無い（既定URL・about:・未確定）ときは通常の Pane ドットへ落ちる。既にナビゲートで
同一 URL の Browser ドットがあればデデュープで畳まれる（二重にならない）。

Pane 記録は `FocusPane`（全プログラム的切替の choke point）とマウスクリックインの `GotKeyboardFocus` の両方で
拾い、デバウンス＋`_trailLastPane` デデュープで 1 点に畳む。`FocusPane` は全ペイン（Debug=IDE 等、内部コントロールが
WPF キーボードフォーカスを取らないものも含む）に確実にフォーカスを入れること。

### §27.3.6 ステージ（ソロ⇄レイアウト）遷移の記録

ソロ（舞台）表示は「対象」ではなく「表示コンテキスト」の変化なので、原則 **Layout 種別 1 ドット**で表す
（Mode／StagePane の変化を含めて記録する）。1 回の操作で余計なドットを積まないこと。

- **ソロ⇄レイアウトのトグル**（タイトルバー／Ctrl+Shift+T → `ToggleDisplayMode`）：`BeginTrailLayoutChange` で
  操作直前を基準にし、`Enter`/`ExitStageMode` 後の保存 choke point（`SaveActiveWorkspaceSnapshot →
  RecordTrailLayoutIfChanged`）で Mode 変化を Layout ドットとして 1 個記録する。
- **舞台ペインの差し替え**（袖・俯瞰カードのクリック → `SetStagePane`）：舞台に立つペインが変わったら、
  その到達を **`Mode=Solo`・`StagePane=新ペイン` の Layout ドット**として 1 個だけ残す。
- **意図**：ソロ表示中に「どのペインを、どの配置で見ていたか」へ戻れること。ジャンプ時の復元は §27.6 の手順
  （`ExitStageMode` → 配置適用 → `Mode==Solo` なら `EnterStageMode(StagePane)`）で行う。

> **実装（2026-07-03）**：Layout ドットの変更検出キーは `mode | stagePane | 構造署名` で作る。構造署名
> （`PaneLayoutTree.StructureSignature`）は比率＝Weight を除外するのでリサイズでは増えず（§27.11-B）、Mode／StagePane を
> 含むのでソロモードでの舞台ペイン切替は独立した Layout ドットになる。判定は保存 choke point
> （`RecordTrailLayoutIfChanged`、同期実行）で行うためデバウンス・フォーカス競合が無く確実。
> **ステージ中は `RecordTrailPane` を記録しない**（`_stageActive` で早期 return）——舞台に立つのは常に 1 ペインで、
> その切替はこの Layout ドットが代表するため、Pane ドットとの二重を避ける（§27.11-G）。

### §27.3.7 ワークスペース切替の記録

- 切替の全域で `_trailSuppressed` を立て、**復元による機械的なタブ活性化・ナビゲート・レイアウト適用を記録しない**。
- 切替元の「離れる瞬間」の配置は、`Trail.SetWorkspace`（ワークスペースキーの切替）**より前に**
  `SaveActiveWorkspaceSnapshot(immediate: true)` で確定させる。後だと復元抑制中で `RefreshLatestTrailPaneLayout`
  が働かず、離脱直前の配置が切替元の最新地点へ残らない。
- 切替後：`SetWorkspace` で表示を今日へ戻して読み直し、ペイン切替デデュープ（`_trailLastPane` 等）を仕切り直す。
  ID 確定後に `EnsureLoaded` を明示的に呼び、復元が長引く間もバーが空のままにならないようにする。
- 各ワークスペースの軌跡は完全分離（§27.8）。切替は「今の作業の続き」ではないので、切替そのものはドットにしない。

## §27.4 現在地と移動（スクラブ）

- **バー上のホイール**：上＝過去（左）、下＝未来（右）へ現在地を 1 つ動かす。端で止まる。
  連続ホイールは 300ms デバウンスで最後の 1 回だけ実ジャンプへ畳む。現在地のドットが見えるよう水平スクロールが追従。
- **ドットのクリック**：その地点を現在地にして即ジャンプ。
- 現在地の移動（ホイール）は表示上の選択を進めるだけで、確定（デバウンス満了）してから実際の画面復元を行う。

## §27.5 ジャンプの直列化

ジャンプは非同期（ファイルを開く等）。実行中に来た新しいジャンプ要求は**中間を捨てて最後の 1 件だけ**処理する
（`_trailPendingJumpEntry`）。ジャンプ処理の全体は `_trailSuppressed` を立てて行い、復元操作が新しいドットを
生まないようにする。処理後は現在地のドットを可視位置へスクロールする。

---

## §27.6 復元（ジャンプ）仕様

ジャンプ 1 件の手順：

1. **事前検証（`CanJumpToTrailEntry`）**：復元が確実に成立するか先に確認し、失敗するなら画面構成を一切変えない
   （「配置だけ変わって対象へ飛べない」を防ぐ）。検証内容：
   - `PaneLayout` があれば deserialize 可能か。
   - File：ファイルが存在するか。／ Browser：target が空でないか。／ Pane：enum 解釈でき実在ペインか。
     / Panel：enum 解釈できるか。／ Terminal：タブ ID が現存するか。／ Session：セッションが現存するか
     （`AiBar.SessionExists`）。／ Layout：`PaneLayout` が非空か。／ Edit：ファイルが存在するか（File と同じ）。
     / Git：**常に false**（ログ専用で戻り先を持たない＝クリックしても復元しない）。
2. **表示コンテキストの復元（`RestoreTrailDisplayContext`、対象より先）**：
   - ステージ中なら一旦 `ExitStageMode`。
   - `PaneLayout` があれば `ApplyPaneLayout`（壊れた 1 件は配置復元だけ省略し、対象へのジャンプは続行）。
   - `Mode == Solo` なら `EnterStageMode(StagePane)`。
3. **対象へ移動（種別ごと、`RegisterTrailJumps`）**：
   - File：ファイルを（無ければ新規タブで）開き、Editor へフォーカス、`line>=0` なら `NavigateTo`。
   - Browser：Browser ペインを可視化＋フォーカスし `NavigateBrowser(url)`。
   - Pane：ペインを可視化＋フォーカスし、`_trailLastPane` を戻り先へ同期。
   - Panel：`ActivePanel` を設定しサイドバーを表示。
   - Terminal：タブが現存すればターミナルを可視化＋そのタブを活性化＋フォーカス。
   - Preview：target のファイルを（無ければ新規タブで）開き、そのタブで EditorSupport ペインを開く
     （`OpenEditorSupportAsync`）。ファイルが消えていれば何もしない。
   - Session：`RestoreSessionById(target)` でセッションを復元し、AI ペインを可視化＋フォーカス。
     削除済みで復元できなければ画面構成を変えず何もしない。
   - Layout：表示コンテキスト復元（手順 2）で完結。個別の対象移動は無い。
   - Edit：File と同じ（ファイルを開き `line>=0` なら `NavigateTo`）。内容は復元しない。
   - Git：戻り先を持たない（手順 1 の `CanJumpToTrailEntry` で弾かれ、ここへは到達しない）。

**戻り先が失われている場合**（ファイル削除・タブクローズ等）は**そっと何もしない**（エラーを出さない。
ブランチ切替等で後から戻ることもあるため記録は消さない）。

## §27.7 日付・時刻ナビゲーション

### §27.7.1 日付ナビゲーション

- 記録は**常に「今日」へ積む**。表示中の日（`DisplayDate`）とは独立に `_todayLatest` を保持する。
- バー左端の日付・時刻ボタンは**日付と時刻を 1 つにまとめた 1 本**（`DateTimeLabel`＝`DateLabel` ＋ `HourLabel`、
  §27.7.2）。**クリックで日付・時刻ポップアップを即トグル**（遅延なし。`StaysOpen=False` の再オープン抑止つき）、
  **右クリックはバー共通メニュー**（「最新に戻る」＝過去日／過去地点から今日の最新へ）。未来日は選べない。
- 過去日を選ぶと読み取り専用の履歴を表示（記録は今日へ積まれ続ける）。`×` ボタンで今日へ戻る。
- **日付跨ぎ（実行中に日が変わる）**：`Today` は都度 `DateTime.Now` から算出する。実行したまま深夜 0 時を
  越えたら、以後の記録は新しい日の DB 行になる。表示・`_todayLatest`・デデュープが古い日に張り付いたままに
  ならないよう追従すること（§27.11-C）。

### §27.7.2 日付・時刻ラベルと 1 つのポップアップ

- 日付と時刻は**1 本のラベル `DateTimeLabel`（＝`DateLabel` ＋ `HourLabel`）**で出す。時刻部分（`HourLabel`）は
  **2 つの表示方法**を持つ：
  - **ライブ（現在時刻）**：表示が今日を追従していて（`_followingToday`）、かつ現在地が軌跡の**最新地点**
    （`CurrentIndex >= Entries.Count - 1`、＝「今」にいる）なら、**現在時刻 `HH:mm`** を出す。時計の進みに
    追従する（View の 30 秒タイマが `RefreshHourLabel` を呼ぶ）。
  - **時間帯（過去）**：過去のドットへスクラブ中、または過去日表示中は、その地点の**時間帯を `HH:00`** で表す
    （`09:00` は **09:00〜09:59** を表す）。
- 現在地が**時間帯（時）を跨ぐと表示も変わる**：`CurrentIndex` の変化で `HourLabel`／`DateTimeLabel` を再評価する
  （ホイールのスクラブ・ドットクリック・時刻選択いずれでも追従）。スクロールできる範囲に時間の制限はない
  （1 日の全ドットを自由に辿れる）。
- **日付と時刻は 1 つのポップアップにまとめる**（旧・別々のカレンダー／時間帯ポップアップを統合）。左に
  **カレンダー**（テーマ追従 `ThemedCalendar`）、右に**その日に記録のある時間帯（`HH:00`）を縦に並べたリスト**
  （`Hours`＝表示中の日の重複なし・昇順の時。記録のある時間が無ければ `HasHours=false` でリストだけ畳む）。
- **右列の縦幅はカレンダーへ固定する**（`DockPanel.Height={Binding ActualHeight, ElementName=TrailCalendar}`）：
  時間帯の記録数でポップアップの縦幅が伸び縮みしないよう、右列の高さをカレンダーに合わせ、時間帯リストが長い日は
  その固定枠の**中でスクロール**させる（旧・`MaxHeight` 指定はリストの数で高さが変わっていた）。
- **右列の先頭に「⟳ 最新に戻る」ボタン**を常時置く（`OnTrailBackToLatestFromPopup`）。過去日表示・過去地点への
  スクラブどちらからも 1 回でライブの最新地点（＝「今」）へ戻す。日付・時間帯の選択と違い、**このボタンは
  ポップアップを閉じてよい**（最新地点を選択扱いにして閉じ、下記のコミットが最新のレイアウトを復元する）。
  時間帯リストが畳まれる（`HasHours=false`）日でもこのボタンは残す。バー共通の右クリックメニュー「最新に戻る」
  （`OnTrailBackToLatest`＝バー内で最新へ動かすのみ）とは、`BackToLatestTrail` を共有しつつ役割が分かれる。
- **選択中はバー内ナビゲーションに留め、画面（レイアウト）の反映は閉じたときにまとめて行う（コミット）**：
  ポップアップは**開いたまま**、日付を選べばその日の軌跡を即バーへ読み込み（時間帯リストもその日の記録へ自動更新）、
  時間帯を選べばその先頭ドットを現在地にしてスクロールする——ここまでは**現在地の移動＋可視化だけ**で画面復元（ジャンプ）は
  しない。**ポップアップを閉じた瞬間**（`TrailDateTimePopup.Closed` → `CommitTrailPopupSelection`）に、選択していた
  現在地の地点へジャンプして画面を復元する。開いてから何も選ばずに閉じたとき（`_trailPopupSelectionMade=false`）は
  画面を変えない。個別のドットを即復元したいときは従来どおり対象のドットをクリックする（§27.6）。
- **選択中の時間帯を強調する**：時刻リストの各ボタンは素はゴースト（透明）で、現在地が属する時間帯
  （`TrailHourViewModel.IsSelected`＝`TrailViewModel` が現在地の変化に追従して更新）だけをアクセント塗り＋太字で
  選択状態に見せる。どの時間帯を見ているのかがリスト上で分かる。
- **ロストフォーカスで閉じる**：選択で閉じないぶん、ポップアップは**フォーカスを失うと閉じる**——枠外への
  キーボードフォーカス移動（`OnTrailDateTimeLostFocus`＝移り先が枠内なら閉じない）と、ウィンドウの非アクティブ化
  （`Deactivated`）、および `StaysOpen=False` による外側クリックで閉じる。開くとき枠へフォーカスを移して仕掛けを効かせる。

### §27.7.3 時間帯の区切り

- ドット列は連続した 1 日の並びのまま、**時間帯（時）が変わる境目に区切りを入れる**
  （`TrailEntryViewModel.StartsNewHour`＝直前のドットと時が異なる先頭。先頭ドットは常に false）。
- 区切りは細い縦線（`|`）だけだと視認しづらいため、**区切り線＋その時間帯の開始時刻ラベル（`HH:00`＝
  `TrailEntryViewModel.HourBandLabel`）を一緒に**出す。時間帯の**先頭ドットの左に、幅 32 の時刻ラベル枠を
  前置**する（非境界のドットでは枠を `Collapsed` にして幅ゼロ＝通常の 14 幅スロットのまま）。
- 先頭ドットのスロットだけ広く（`20 + 32`）なるため、スクロール位置を出す `TrailSlotOffset` は**等間隔ではなく
  各ドットの実効幅を積んで**現在地スロットの左端を求める（`TrailDotWidth=20` / `TrailHourLabelWidth=32` を XAML と揃える）。

### §27.7.4 スクロール（左端の決め方）

- スクロール追従は **1 関数 `ScrollTrailToCurrent` に集約**する：現在地ドットのスロット左端
  （`TrailSlotOffset(CurrentIndex)`）を表示領域の左端へ寄せるだけ。最新地点なら `ScrollViewer` のクランプで
  そのまま右端に収まる＝**ライブの「最新を右端に張り付ける」も、途中地点の「その地点を左端に置いて以後の並びを
  右に見せる」も同じ計算**で表す（中央寄せ・末尾余白は廃止）。
- **追従するか否か**は真偽値 `_trailBrowsingPast` 一つで決める：ユーザーが過去を見ている間 true で、追記が来ても
  左端を動かさない（レイアウト変更などで最新へ引っ張らない）。過去のドットへのジャンプ／スクラブ／時間帯指定で
  非最新へ移ると true、最新へ移動・「今」ダブルクリック・素ホイールで右端まで戻ると false（ライブ追従）へ戻す。

## §27.8 ワークスペース分離

軌跡はワークスペース（`workspace` 列＝`WorkspaceSnapshot.Id` 文字列、未オープンのスクラッチは空文字）ごとに
完全分離する。混ざると別ワークスペースのファイル／タブへ飛んで破綻するため。ワークスペース切替時は表示を今日へ
戻して読み直し、ペイン切替デデュープ状態（`_trailLastPane` 等）も仕切り直す。切替元の「離れる瞬間」の配置は
`SetWorkspace` の**前に**確定させる。

## §27.9 永続化（SQLite / `TrailStore`）

- 保存先：`%APPDATA%/Loomo/trail.db`（`LOOMO_TRAIL_DB` で上書き可）。WAL、接続は開きっぱなし。
  すべて UI スレッドからの小さな読み書き。失敗時は呼び出し側が握りつぶし**メモリ内動作へ縮退**する
  （軌跡バーは動き続ける）。
- テーブル：
  - `trail_entries(id, workspace, day, timestamp, kind, target, label, line, col, display_mode, stage_pane, layout_id)`。
    `day` はローカル日付（`yyyy-MM-dd`）。index は `(workspace, day, id)`。
  - `trail_layouts(id, snapshot UNIQUE)`：同一配置 JSON は 1 行だけ保持し、各エントリが `layout_id` で参照。
    参照が切れた配置行はデデュープ更新・配置更新時に回収する。
- スキーマは加算移行（`user_version`、不足列は既定値付き `ALTER`）。既存行を破棄しない。
  DB のスキーマが本体より新しい場合は開かずに例外。
- 更新 API：`Append` / `Update`（デデュープ：時刻・ラベル・位置・配置）／ `UpdatePosition`（離脱位置のみ）
  / `UpdatePaneLayout`（配置のみ）。読み取り：`LoadDay` / `HasAny` / `ListDays`。

## §27.10 UI 仕様

- ウィンドウ最下部（ShellLayout Row 2）の全幅バー、高さ 26。バーの表示可否は **`BarVisible`＝`HasEntries`（記録が
  1 件でもある）かつ `Visible`（ユーザー設定による ON/OFF）** の両方で決まり、どちらかが偽ならバーごと非表示で高さを
  取らない。
- **表示 ON/OFF（`Visible`）**：ユーザーが軌跡バーを隠せる。値は `AiSettings.TrailVisible`（既定 true）へ永続化する。
  切り替え口は 2 つ：(1) 設定オーバーレイ「外観」の「軌跡バーを表示する」チェックボックス（`Trail.Visible` に双方向
  バインド）、(2) **バーを右クリック → コンテキストメニュー「軌跡を非表示にする」**（`Trail.HideCommand`＝`Visible=false`）。
  OFF でも記録自体は続くので、再表示すればそれまでの軌跡も見える。再表示は設定トグルからのみ（隠れている間はバーを
  右クリックできないため）。子ボタンは独自のコンテキストメニューを持たず、右クリックはバー共通メニューへ遡る。
- 左：日付＋時刻ボタン 1 本（`DateTimeLabel`、`OnTrailDateTimePreviewDown`）→ **日付・時刻ポップアップ**
  （カレンダー〔テーマ追従 `ThemedCalendar`〕＋右列を横並びで内包。右列は縦幅をカレンダーへ固定し、先頭に
  「⟳ 最新に戻る」ボタン〔`OnTrailBackToLatestFromPopup`〕、その下に `HH:00` を縦に並べた時間帯リストを置く。
  `OnTrailCalendarSelected` でその日を、`OnTrailHourSelected` でその帯へ、いずれもポップアップを開いたまま即反映、§27.7.2）。
  右：`×`（過去日表示中のみ）。中央：ドット列（`ScrollViewer`＋`ItemsControl`、水平のみ、ホイールは現在地移動へ）。
- **時間帯の区切り**：ドットのスロット左端に、時間帯が変わる先頭（`StartsNewHour`）でだけ幅を持たない細い
  `|`（`Rectangle`）を出す（§27.7.3）。
- 各地点は幅 20 のスロットに **`IconGeometry`（種別の絵姿）を `Path` で描く**（丸＝Ellipse ではない）。
  アイコンは**単色ベクターの線画ピクトグラム**（16×16 座標・塗り無し・絵文字を使わない）＝`Stroke` に前景ブラシを張り、
  テーマ追従と現在地の色強調が効く。抽象記号ではなく「何の地点か」がパッと見で分かる絵姿で、種別ごとに一意
  （File＝書類／Browser＝地球儀／Terminal＝プロンプト `>_`／Preview＝目／Session＝吹き出し／Edit＝鉛筆／
  Git＝ブランチ／Pane＝二分割ウィンドウ／Panel＝サイドバー／Layout＝2×2 グリッド）。線画ピクトが読める最小は
  16px＝行の縦幅もこれで決まる（12px 以下は破綻するため縦を詰めすぎない）。
  色は 4 カテゴリに集約：内容(File/Browser/Terminal/Preview)=Fg、変更(Edit/Git)=Success、構造(Pane/Panel/Layout)=FgDim、
  AI(Session)=Accent。現在地はアクセント色で `StrokeThickness` を上げ 1.18 倍に拡大。抽象記号の `Glyph` はツールチップ
  本文と種別一意性テスト用に併存する（バーの描画は `IconGeometry`）。
  `ToolTip`（種別・名前・対象・モード・日時）と `AutomationProperties.Name`＝`AccessibleName`（スクリーンリーダー用）も持つ。
- スクロール追従の `dotWidth=20` は XAML のスロット幅と 1 箇所で揃える（重複定数の同期ずれに注意）。

## §27.11 既知の不具合と修正状況

**A〜D・G は 2026-07-03 修正済み**（`TrailViewModelTests` / `PaneLayoutStructureSignatureTests` で担保）。

- **A（高・記録が止まる）〔修正済〕**：`RecordTrail` が `!_trailLayoutTrackingReady` で早期 return し、
  `_trailLayoutTrackingReady` は `BeginTrailLayoutChange`（レイアウト変更の choke point）でのみ true になり
  **起動・ワークスペース復元経路（`LoadLayouts`）では立たない**ため、セッション開始後に最初のレイアウト操作を
  行うまで File/Browser/Pane/Panel/Layout の記録が一切積まれなかった。→ `_trailLayoutTrackingReady` を廃止し、
  `RecordTrail` のゲートを `_trailSuppressed` のみにした。起動・復元は `_trailSuppressed` が全域（遅延
  ハイドレーションの `await Dispatcher.Yield` も抑制領域内）でカバーするため、記録は起動直後から効く。
- **B（中・ノイズ）〔修正済〕**：Layout の同一判定 `CurrentTrailLayoutState` が配置 JSON **全体（Weight 含む）**を
  ハッシュ化していたため、ペインのリサイズごとに Layout ドットが増えた。→ 変更検出キーを
  `mode | stagePane | PaneLayoutTree.StructureSignature`（Weight は除外、構造・モード・舞台ペインは反映）に置換。
- **C（中・日付跨ぎ）〔修正済〕**：深夜 0 時を越えて実行すると `DisplayDate`/`_todayLatest` が前日に張り付いた。
  → `TrailViewModel` に時計注入（`Func<DateTime>`）と `_liveDay`/`_followingToday` を導入し、`Record` 冒頭の
  `RollLiveDayIfNeeded` で日跨ぎを検知（追従中は表示を新しい今日へ繰り上げ・デデュープ仕切り直し）。
  `IsViewingPast` は追従状態で判定する。
- **D（低・視認性）〔修正済〕**：バーは `Glyph` を描かず**色分けした丸**を出しており、しかも File・Terminal・Layout に
  色/形の区別が無く**全部が同じ丸**に見えていた（ユーザー報告「記号が全部ドット」）。→ バーの各地点を
  `Glyph`（単色の幾何記号）を描く `TextBlock` に置換し、種別ごとに一意の記号＋色分け＋現在地強調にした（§27.10）。
  記号も一貫した単色セット（`◆◉▦◫❯⊞`）へ整えた。
- **G（中・二重ドット）〔修正済〕**：ステージ遷移で Pane ドットと Layout ドットが二重に積まれていた
  （`SetStagePane`／`EnterStageMode` の Pane 記録と、保存が起こす `RecordTrailLayoutIfChanged` の Layout 記録）。
  → ステージ中は `RecordTrailPane` を記録しない（`_stageActive` で早期 return）ようにし、舞台ペインの切替は
  保存 choke point の Layout ドット 1 個で代表する。`SetStagePane` の（デバウンスされる）Pane 記録呼び出しは撤去。
- **H（回帰・修正済）**：B/G の初回修正でキーから Mode/StagePane を外した際、**ソロモードでの舞台ペイン切替が
  一切記録されなくなった**（Layout ドットは出ず、ステージ中の Pane 記録はデバウンス・フォーカス競合で不発）。
  → キーに Mode/StagePane を戻し（Weight のみ除外）、保存 choke point の確実な Layout ドットで代表するよう修正。
- **E（低・将来課題／未対応）**：ブラウザの「戻る」は到着が `NavigationCompleted` 経由で再記録される非対称。
  出自（何から遷移したか）は未記録（§27.1 の非目的だが Thread Rail の将来課題）。
- **F（確認済み・解消方向）**：`RecordTrailPane` はタイマ確定時に `_trailLastPane` を更新するが記録は
  `RecordTrail` 経由。A の修正で抑制ゲートが `_trailSuppressed` のみになり、起動直後にデデュープ状態だけ進んで
  記録されない取りこぼしは解消。
