# エージェントループ知見

エージェントループ（`AgentOrchestrator`）と Ollama ローカル推論まわりで、調査して分かったことを貯めるドキュメント。
「設計の意図」は `設計書.md` と `CLAUDE.md`、ここは**実測ベースの知見・落とし穴・診断手順**を書く。新しい発見は追記する。

---

## 1. 遅延の正体：ローカルLLMはほぼ prefill 律速（CPU実行時）

### 計測のしかた
- 各AI呼び出しの内訳は `ai.usage` トレースに記録される（`%APPDATA%/Loomo/traces/*.jsonl`）。
  - トークン数：`inputTokens`(=Ollama `prompt_eval_count`) / `outputTokens`(=`eval_count`)
  - 段階別所要(ms)：`loadMs`(重みロード) / `promptEvalMs`(prefill) / `evalMs`(decode) / `totalMs`
  - 実装：`OllamaProtocol.ParseUsage`（done行のナノ秒を ms 化）→ `AiUsageReported` → `AgentOrchestrator` が記録＆UI へ流す。
- UI では AIバーの進行状況に「📊 AI内訳#N: トークン … ｜ ロード… prefill… decode…」として出る（`AiBarViewModel.FormatUsage`）。
- サーバ側の配置は `ollama ps`（CPU/GPU 比率・CONTEXT・常駐期限）、GPUは `nvidia-smi`。

### 実測例（2026-06 / phi4-mini:latest 3.8B / num_ctx 8192 / CPU実行）
| 状況 | 入力tok | prefill | decode | load |
|---|--:|--:|--:|--:|
| 「pwd」turn1 iter1 | 471 | 18.9s | 1.4s(出15) | 0.28s |
| 同 turn1 iter2(ツール往復後) | 526 | 8.8s | 1.6s(出17) | — |
| 圧縮後 別セッション turn1 | 383 | 18.6s | 12.7s(出127) | 0.31s |
| 連続 turn1 | 383 | 18.7s | 9.3s(出99) | 0.29s |
| 連続 turn2 | 486 | **36.5s** | 10.2s(出105) | 0.29s |

### 結論
- **遅さの主因は prefill（プロンプト評価）。** decode/load ではない。CPUで prefill ~13–20tok/s、decode ~10tok/s。
- `loadMs` は ~0.3s ＝ 重みは常駐済み（`keep_alive=-1` は効いている）。18秒は純粋な prompt-eval 計算時間。
- **prefill 時間はトークン数にきれいに比例しない**（471→383 に減らしても 18.9→18.6 とほぼ不変）。
  「per-token は速いが固定費が大きい」のではなく、後述のとおり**cold/warm（キャッシュ有無）と会話長**が支配的。

## 2. ターン間プレフィックスKVキャッシュは“当てにならない”（この環境での実測）

- 期待：turn2 は `[system][tools][user1][assistant1]` を再利用し、user2 ぶんだけ prefill → 数秒。
- 実測：**turn2(36.5s) は turn1(18.7s) より遅い**。会話が伸びるほど prefill が増え、毎ターン全文を再評価している signature。
- 同一ターン内の iter2（ツール結果を即フィードバック）は速かった（8.8s）が、**別ユーザーターンをまたぐと効かない**。
- キャッシュを壊しうる既知要因（コードに内在）：
  - 〔解消済〕以前は `WorkspaceContext.Describe` の現在フォルダ文脈を**毎ターン最新 user メッセージ末尾へ**付け直して
    いたため、turn1 で user1 に付いた文脈が turn2 では user2 へ移り、**user1 のバイト列がターンで変わって会話履歴ぶんの
    再利用が途切れて**いた。現在はワークスペースルートを system プロンプト（`OllamaPromptBuilder.Build`）に含めるように
    したので、user メッセージは安定し、この要因は解消（ルートはフォルダを開き直したときだけ変わるので prefix も準安定）。
  - 起動時ウォームアップ（`LocalLlmWarmupService`→`WarmChatPrefixAsync`）で温めた `[system][tools]` プレフィックスが、
    最初の実ターン(turn1 cold 18.7s)に**繋がっていない**ように見える。要因候補：warmup 完了前の送信／
    `WarmModelAtStartupAsync` の `/api/generate prompt:""` と `WarmAsync` の競合でキャッシュ上書き／`RootChanged` 連発による前 warmup のキャンセル。
- 教訓：**ローカルCPU推論ではプレフィックスキャッシュ前提の最適化は過信しない。** 効けば儲けものくらいの位置づけ。

## 2.1 【ONNX移行後の更新】プレフィックスKVキャッシュは“効く”ようになった

- §2 は **Ollama 構成での実測**。ONNX Runtime GenAI への移行（[[onnx-genai-engine]]）後は事情が変わった。
  `Phi4Engine` が**常駐 `Generator` ＋ `_fedTokens`** を持ち、`PrepareKvCache` が前回フィード列と今回プロンプトの
  **最長共通接頭辞を `RewindTo`＋`AppendTokens` で再利用**する（分岐分だけ再 prefill）。さらに `PrimeAsync`
  で起動時に `system＋tools` を暖機する。`Phi4PromptFormatter` の tools ブロックは会話の前＝安定プレフィックス内。
- 結果：**安定プレフィックス（system＋tools＋既存履歴）の prefill は初回/暖機の 1 回に償却**され、turn2 以降は
  再利用される。Ollama 時代の「毎ターン全文を再 prefill」は ONNX では当てはまらない。
- **ツール設計への含意**：「だから 1 ツール（旧 §3.5/3.6 や CLAUDE.md の根拠）」は弱まった。`run_powershell` に
  加えて `write_file`/`edit_file` を追加済み（2026-06）。ツール定義の prefill は償却されるので、数個の追加は
  毎ターンの負担にならない。**残るコストは（a）ツール選択の信頼性＝反復回数**（KVキャッシュは救わない・§3.5）
  **と（b）キャッシュ無効化イベント**（モデル再ロード／`max_length`・sampling 変更で `Generator` 再構築＝KVクリア／
  フォルダ root 変更は system に root を含むため tools ブロックも再 prefill）。ツールは少数・排他的・動詞名に保つ。
- **計測の指針**：ツールを足したら `ai.usage` で（1）turn2+ の prefill が低いまま（tools がキャッシュ再利用されている）
  と（2）平均イテレーション数が増えていない、を固定タスク集で確認してから本採用する。

## 3. 真因：GPUが非力でCPU実行に落ちている（ハード起因）

- `ollama ps` → `phi4-mini:latest 3.7GB  PROCESSOR 97%/3% CPU/GPU  CONTEXT 8192`。
- GPU は **GeForce GT 710 / VRAM 1GB**。3.7GB のモデルが載らず 97% を CPU にオフロード。
  GT 710 自体も2014年の最安品でLLM推論には実質無力（テンソルコア無し・帯域 ~14GB/s）。
- これで全部説明がつく：prefill/decode の遅さ、プロンプト圧縮がほぼ効かない（壁は**計算量**でトークン数でない）、
  キャッシュ小細工が救えない。

## 3.5 反復回数 × prefill が支配的 — tool call の一発成功率が効く

- エージェントループは **1イテレーション＝AI 1呼び出し＝prefill 1回**。CPUでは prefill が数十秒なので、
  **tool call が失敗して往復が増えると、その回数ぶん丸ごと prefill を払い直す**。
- 実測（qwen3:1.7b・モデル切替直後）：pwsh が2回エラー→AI 3往復で合計 1m15s。内訳は
  iter#1(load 7.5s + prefill 31.9s) / iter#2(prefill 16.5s) / iter#3(prefill 8.5s)。**遅さの主因は反復回数**。
- 教訓：**「速いが間違える小型モデル」より「やや遅いが一発で当てるモデル」のほうが総時間で勝つことがある。**
  モデル選定は decode tok/s だけでなく **tool call の一発成功率（=平均イテレーション数）** で評価する。
  `ai.usage` を見るときは各ターンの **iter 数**（📊AI内訳#N の最大N）も必ず確認する。
- 補足：ターン内（ツール往復）のプレフィックスキャッシュは効くことを確認（prefill が iter を追うごとに減る）。
  ただし**初回 iter#1 のコールド**（モデル切替直後はディスクフォルトインで特大）は別途大きい。
  モデルを変えたら**warm 状態で再測定**しないとフェアな比較にならない（切替直後の1発目は load+cold で過大評価）。

## 3.6 モデル別の実観測（tool 一発成功率）

- **キャッシュが warm なら prefill は爆速**：qwen3:1.7b で 434トークンのプレフィックスが **prefill 193ms**（≈2250tok/s）で
  再利用され、warmターン1発目が合計2.3秒。過去の phi4-mini「18秒」級はコールド/未プライムの値だった。
  → **モデル比較は必ず warm かつ「同じ問いを一発で解けるか（イテレーション数）」で見る。**
- **qwen3:1.7b は tool/コマンド生成が弱い（実測・要注意）**：「pwd」に対し
  `Get-ItemPropertyValue ... -Property 'pwd'` → `Get-Item ... -Recurse` → `Get-ChildItem ... -Recurse` と
  3回連続で的外れ（正解は `Get-Location`/`pwd`）。速いが外すため総時間でもUX的にも不利。
- **phi4-mini(3.8B) は同じ「pwd」を一発で `Get-Location`**。decode は遅いが一発成功で総時間は安定。
- 指針：このエージェント（pwsh 単一ツールで全部やる設計）では **3B以上＋コマンド/コード適性**が無難。
  候補は **qwen2.5-coder:3b**（コマンド生成に強い）や phi4-mini。1.7B以下は速いが tool 失敗の再prefillで相殺。

## 4. 効く対策（効果順）

1. **モデルを小さくする（無料・即効・最有力）。** CPU速度はパラメータ数にほぼ反比例。
   3.8B→1.7B で約2倍、→0.6B で約6倍。prefill も decode も同時に効く。トレードオフは tool call 安定性。
   - 2026-06 時点でインストール済み：phi4-mini(3.8B)/qwen3:4b/qwen3:1.7b/qwen3:0.6b/qwen2.5:3b/qwen2.5-coder:3b/gemma3:4b(tools非対応)。
   - **採用：qwen3:1.7b**（tools+thinking 対応、速度と賢さの折衷）。`settings.json` の `local.model` を変更。
2. **GPU増設（VRAM 8GB以上）。** phi4-mini をフルGPUに載せれば prefill は桁違い。ハード投資。
3. **コンテキストを切り詰める。** 毎ターン全文 prefill されるので、会話が伸びると線形以上に悪化（turn2=36.5s）。
   `num_ctx` とトリム上限（`SettingsContextWindowPolicy`）で上限を抑えると肥大を緩和できる。

## 5. プロンプト圧縮の効果（参考・限定的）

- 安定プレフィックス（system＋tools＋style＋search）を **388→約278トークン（約28%減）** に圧縮（コミット `575794e`）。
  - system本体 ~230→~176（導入文1行化・`SafetyPolicy` で強制済みの安全文言削除等）
  - phi4-mini styleGuidance ~37→~19（system規則と重複する指示を削除）
    → **その後 styleGuidance の仕組みごと撤去**。モデル固有のプロンプト差し込みはしない方針に変更し、
      システムプロンプトは全モデル共通に統一（`ModelProfile.StyleGuidance` 削除）。
  - pwsh ツール説明文 ~63→~24（「ファイル操作も全てpwsh」は system 側に集約済み）
- **システムプロンプトは英語指示／日本語出力に変更**。小型ローカルモデルは英語の tool-calling 指示の方が
  従いやすい傾向があるため（qwen3:1.7b の「pwd」誤り等の対策の一環）。最終回答だけ日本語を指示。
- 効果は限定的：**初回(cold)とモデル再ロード時に効くだけ**で、CPU律速の壁は越えない。
  トークン削減より**モデルサイズ削減**のほうが桁で効く。

## 5.5 評価ハーネスとプロンプトの「共進化＝過適合」に注意（2026-06）

- 能力ハーネス（`AgentCapabilityHarness`）の固定15タスクに対し、システムプロンプトを
  「落ちたタスクを直す1行を足す→R=3で測る」のループで改良し続けた結果、**プロンプトと評価セットが共進化**した。
  few-shot 例にハーネスのシードファイル名（`Read README: … Get-Content README.md`）が混入し、
  rename カーブアウト・edit_file 明示行など**個別タスクの対症文**が積み上がった。
- この状態で測った「プロンプト変更はゼロ和（Pareto シーソー）」という結論は、**チューニングに使った
  のと同じタスク集合で測った値**であり、汎化性能の証拠にはならない（飽和した過適合評価では、
  どんな変更もヘッドラインを動かさず「ゼロ和」に見える）。
- 対策（2026-06 実施）：
  1. **評価セットの拡張**：プロンプトの例と語彙・ファイル名・操作が被らない10タスクを追加（25タスク化）。
     コピー/削除/最大値/件数/横断検索/全置換/JSONキー追加/意味的バグ修正など。
  2. **プロンプトの汎用化**：ハーネス固有名を全廃し、対症文を一般原則（事実はツール結果からのみ・
     エラーを成功と報告しない・複数部の完遂と検証・old_string 厳密複写）へ昇格（`Qwen3SystemPrompt`）。
  3. **再混入の機械的防止**：`Qwen3PromptFormatterTests` がシードファイル名の混入を assert で拒否。
- 教訓：**評価セットはプロンプトと直交に育てる**。プロンプトに足した語がそのまま評価タスクの語彙に
  ある場合、その PASS は能力でなく暗記。「能力はコードに」の原則は維持しつつ、プロンプト側の効果測定は
  チューニングに使っていない held-out タスクで行う。

## 5.6 ターミナル基盤の3故障＝「読めないエラー・stdin ハング・子プロセスリーク」（2026-06）

ハーネス唯一の hard-fail（find-text 0/3）の調査で、ターミナル実行基盤に**モデルの能力と無関係な**
故障が3層重なっていることが判明した。いずれも決定論的なインフラ修正で解消（find-text 0/3→3/3、
全25タスク R=3 で **全試行PASS 25/25**＝旧最良21/25から底上げ・回帰なし）。

1. **stderr が cp932/UTF-8 不整合で文字化け**：Windows PowerShell 5.1 はエラーを cp932 で書くが、
   読み手のデコードと不整合で `�p�X...` 状に潰れ、**モデルは失敗理由を一切読めず自己修正不能**だった。
   → `TerminalService` がコマンドに UTF-8 固定の前置句（`[Console]::OutputEncoding`/`$OutputEncoding`）を
   付け、`StandardOutput/ErrorEncoding=UTF8` と整合させた。エラーが読めるようになるだけで
   **全タスクの自己修正力が底上げ**される（25/25 の主因のひとつ）。
2. **rg の stdin 永久ブロック**：`rg --files <語>` のような誤用でパス引数が消えると、rg は
   （stdin が TTY でないため）**stdin を読みに行き永久に待つ**。旧ハーネスは stdin を閉じておらず、
   毎回5分タイムアウトで FAIL（痕跡＝300008ms/iters少）。→ stdin をリダイレクトして即クローズ
   （EOF）。誤用コマンドは即座に exit 1 で返り、モデルが反復内で挽回できる。
3. **キャンセル時の子プロセスリーク**：タイムアウトで `WaitForExitAsync(ct)` が投げても子を kill
   しないと rg/powershell がリークし、**継承した stdout ハンドルで testhost のパイプを握って
   `dotnet test` ごとハング**させる（前夜の rg が翌朝まで生存していた）。→ キャンセル時に
   `Kill(entireProcessTree: true)`。長時間ラン前は `Get-CimInstance Win32_Process` で残骸確認。

- 実装は**ハーネス専用コードを捨てて実 `TerminalService` をそのまま使う**形にした（HeadlessTerminal
  廃止）。評価が実機と同一挙動になり、この種の「ハーネスだけの故障」を構造的に防ぐ。
- 残ったモデル挙動（`rg --files` をパターン検索と誤解し、読めるエラーでも「該当なし」と誤読）は
  既存の決定論ヒント機構で対処：`--files`＋`rg: …(os error 2)` を検知して「--files はパターンを
  取らない。外して再実行（例: 修正済みコマンド）」を機械的に提示（`--files-with-matches` は除外）。
  **インフラ修正だけでは find-text は 0/3 のままで、ヒントを足して 3/3**＝「エラーが読める」と
  「次の一手の機械的提示」の両方が要る。

## 6. 診断クイックリファレンス

- 「遅い」と言われたら、まず進行状況の **📊 AI内訳** か `traces/*.jsonl` の `ai.usage` で **prefill / decode / load** を切り分ける。
- `load` が大きい → 重みが常駐していない（`keep_alive`・モデル再ロード・初回）。
- `prefill` が大きい → プロンプト評価。CPU実行・大きいモデル・長い会話を疑う。`ollama ps` で CPU/GPU 比を見る。
- `decode` が大きい → 出力トークンが多い（冗長生成）か、CPUの decode 速度。`outputTokens` を見る。
