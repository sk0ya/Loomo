# AI Agent Studio 設計書

> アプリ名: **Loomo**（ルート名前空間: sk0ya.Loomo）
> 作成日: 2026-05-31

---

## 1. 概要

ローカルの開発ワークスペースを対象に動作する **AIエージェント搭載のデスクトップアプリ**（C# / WPF）。
ユーザーは自然言語で指示を出し、AIエージェントが **Terminal（コマンド実行）** と **Editor（ファイル表示・編集）** を操作しながらタスクを遂行する。

### コアコンセプト
- AIエージェントは UI を「人間と同じ道具」として使う。すなわち FolderTree で対象を把握し、Terminal で実行し、Editor で表示・編集する。
- UI の各ペインは **疎結合なサービス + メッセージング** で連携し、AIエージェントは「ツール（Tool）」を通じてそれらを駆動する。
- モダンな見た目（ダーク基調・フラット・VS Code 系の操作感）と、機能を後から足しやすい拡張構造を両立する。

## 2. 技術スタック

| 領域 | 採用 | 理由 |
|------|------|------|
| ランタイム | .NET 8 (LTS) もしくは .NET 9 | WPF 安定サポート。LTS優先なら .NET 8 |
| UIフレームワーク | WPF | 要件指定。リッチなデスクトップUIに最適 |
| アーキテクチャ | MVVM | テスト容易・View/ロジック分離 |
| MVVM基盤 | CommunityToolkit.Mvvm | `ObservableObject` / `RelayCommand` / `WeakReferenceMessenger` |
| DI / 起動 | Microsoft.Extensions.Hosting + DependencyInjection | サービス登録・ライフサイクル管理・設定・ロギングを統一 |
| ロギング | Microsoft.Extensions.Logging（+ Serilog 任意） | 構造化ログ |
| Terminal | **sk0ya.Terminal** (NuGet) | 要件指定 |
| Editor | **sk0ya.Editor** (NuGet) | 要件指定 |
| AIクライアント | Anthropic Claude API（Tool Use対応） | エージェントのツール実行に最適。`IAiClient` で抽象化し差し替え可能に |
| アイコン | 任意（Lucide / Fluent Icons の Path 化） | モダンな見た目 |

> AIプロバイダは `IAiClient` 抽象の背後に隠すため、Claude 以外（OpenAI等）への差し替えやローカルLLM対応も後から可能。

## 3. 全体アーキテクチャ

レイヤー構成（依存は上→下の一方向）:

```
┌─────────────────────────────────────────────────────────┐
│  Presentation (WPF)                                       │
│   Views (XAML) ── ViewModels ── Behaviors / Converters   │
└───────────────┬─────────────────────────────────────────┘
                │  Messenger（イベント） / DI
┌───────────────▼─────────────────────────────────────────┐
│  Application / Agent Core                                 │
│   AgentOrchestrator（エージェントループ）                 │
│   ToolRegistry / IAgentTool 群                            │
│   ApprovalService（実行承認）/ ConversationStore          │
└───────────────┬─────────────────────────────────────────┘
                │  サービス抽象（インターフェース）
┌───────────────▼─────────────────────────────────────────┐
│  Services / Adapters                                      │
│   ITerminalService → sk0ya.Terminal                      │
│   IEditorService   → sk0ya.Editor                        │
│   IWorkspaceService→ ファイルシステム / FolderTree状態     │
│   IAiClient        → Claude API                          │
└──────────────────────────────────────────────────────────┘
```

### 連携の要点
- **View 同士は直接参照しない。** すべて `WeakReferenceMessenger` のメッセージ、または共有サービス経由でやり取りする。
- AIエージェントは UI を直接触らず、**ツール → サービス → （メッセージ）→ View** という経路で操作する。これにより「AIが操作する」「人間が操作する」が同じ経路に集約され、テスト・記録・再現が容易になる。

## 4. UIレイアウト（確定案）

採用: **全幅の下部AIバー（展開式）**

```
┌────┬──────────┬───────────────┬────────────────┐
│    │          │               │                │
│ A  │  Folder  │   Terminal    │     Editor     │
│ c  │  Tree    │  (sk0ya.      │   (sk0ya.      │
│ t  │ (Sidebar)│   Terminal)   │    Editor)     │
│ i  │          │               │                │
│ v  │          │               │                │
│ t  │          │               │                │
│ y  │          │               │                │
├────┴──────────┴───────────────┴────────────────┤
│ 🤖 AIに指示を入力...              [▲展開] [送信] │  ← 通常時：1行バー（全幅）
└─────────────────────────────────────────────────┘

  ▲展開 すると：

├─────────────────────────────────────────────────┤
│ ● 会話履歴 / ストリーミング出力                   │
│   - ツール実行カード（コマンド・差分・承認ボタン）│   ← 展開時：全幅パネル
│   - エージェントの思考/結果                       │
│ ─────────────────────────────────────────────── │
│ 🤖 AIに指示を入力...              [▼閉じる][送信] │
└─────────────────────────────────────────────────┘
```

### 各領域
| 領域 | 内容 | 幅/高さ |
|------|------|---------|
| ActivityBar | アイコン縦並び（エクスプローラ/検索/AIセッション/設定 等）。サイドバー内容を切替 | 固定 48px |
| Sidebar | 既定は FolderTree。ActivityBarの選択で内容差替（拡張ポイント） | 可変・GridSplitter |
| Terminal | sk0ya.Terminal をホスト。複数タブ対応可 | 可変・GridSplitter |
| Editor | sk0ya.Editor をホスト。複数タブ（ファイル）対応 | 可変・残り全幅 |
| AIバー | 下部全幅。通常1行／展開でパネル化（GridSplitterで高さ調整） | 既定 48px ⇔ 展開時 ~35% |

### レイアウト実装方針
- ルートは `Grid`（行: メイン領域 / AIバー、列内はネストGrid）+ `GridSplitter` でリサイズ可能に。
- 後述の「パネル登録方式」により、各セルの中身を直接ハードコードせず `ContentControl` + ViewModel で差し込む。
- AIバーの展開/縮小は ViewModel の状態 + `RowDefinition` の高さアニメーションで表現。
- 配色・角丸・余白は `ResourceDictionary`（テーマ）に集約し、ダーク/ライト切替に対応。

## 6. 主要コンポーネント設計

### 6.1 サービス抽象（Abstractions）

```csharp
public interface ITerminalService
{
    Task<CommandResult> RunCommandAsync(string command, CancellationToken ct);
    void Write(string text);                 // 入力流し込み
    event EventHandler<string> OutputReceived;
    Task CancelAsync();
}

public interface IEditorService
{
    Task OpenFileAsync(string path);
    Task<string> GetActiveContentAsync();
    Task ShowDiffAsync(string path, string proposedContent);   // 差分プレビュー
    Task<bool> ApplyEditAsync(string path, string newContent); // 適用（要承認）
    string? ActiveFilePath { get; }
}

public interface IWorkspaceService
{
    string? RootPath { get; }
    string? SelectedPath { get; }            // FolderTreeの選択
    Task<IReadOnlyList<FileNode>> ListAsync(string path);
    event EventHandler<string?> SelectionChanged;
}
```

これらの **実装** は `Loomo.Services` に置き、sk0ya.Terminal / sk0ya.Editor のコントロールをラップする。
（sk0ya.* の実APIに合わせてアダプタ内で吸収。API差異の影響をこの層に閉じ込める。）

### 6.2 AIクライアント

```csharp
public interface IAiClient
{
    IAsyncEnumerable<AgentEvent> StreamAsync(
        Conversation conversation,
        IReadOnlyList<ToolDefinition> tools,
        CancellationToken ct);
}
// AgentEvent = TextDelta | ToolUseRequested | TurnCompleted ...
```

ストリーミングで `TextDelta`（逐次テキスト）と `ToolUseRequested`（ツール呼び出し要求）を流す。

## 7. AI連携設計（最重要）

### 7.1 ツール（Tool）= AIの手足

AIエージェントは Claude の **Tool Use（function calling）** を使い、以下のツールを呼び出す。各ツールはサービスを叩くだけの薄いラッパ。

| ツール名 | 接続先サービス | 説明 | 承認 |
|----------|----------------|------|------|
| `list_directory` | IWorkspaceService | フォルダ内容の取得 | 不要 |
| `read_file` | IEditorService/FS | ファイル読み取り | 不要 |
| `open_in_editor` | IEditorService | Editorで開く | 不要 |
| `propose_edit` | IEditorService | 差分を提示（適用は承認後） | **必要** |
| `run_command` | ITerminalService | Terminalでコマンド実行 | **必要** |
| `get_selection` | IWorkspaceService | FolderTreeの現在選択を取得 | 不要 |

```csharp
public interface IAgentTool
{
    string Name { get; }
    ToolDefinition Definition { get; }              // JSON Schema（Claudeへ渡す）
    bool RequiresApproval { get; }
    Task<ToolResult> ExecuteAsync(JsonElement args, CancellationToken ct);
}
```

新しい能力の追加 = `IAgentTool` を実装して DI 登録するだけ（**拡張ポイント**）。

### 7.2 エージェントループ（AgentOrchestrator）

```
ユーザー入力（下部AIバー）
        │
        ▼
[1] Conversation に user メッセージ追加
        │
        ▼
[2] IAiClient.StreamAsync(会話, ツール定義)
        │
        ├── TextDelta ───────────► AIバーに逐次表示
        │
        └── ToolUseRequested
                │
                ▼
        [3] RequiresApproval?
                ├─ Yes → ApprovalService → AIバーに承認カード表示 → ユーザー判断
                └─ No  → そのまま実行
                │
                ▼
        [4] ToolRegistry.Get(name).ExecuteAsync(args)
                │  （= Terminal実行 / Editor操作 / FS読取）
                ▼
        [5] ToolResult を Conversation に追加 → [2] へ戻る
        │
        ▼
最終テキストで完了（TurnCompleted）
```

- ループ・状態は `Loomo.Core` 側に持たせ、UIはイベント購読のみ。
- 中断（キャンセル）対応：`CancellationToken` を全層に伝播。

### 7.3 シーケンス例：「このプロジェクトをビルドしてエラーがあれば直して」

```
User → AIバー：「ビルドしてエラーを直して」
Agent → run_command("dotnet build")        [承認] → Terminalで実行
Terminal → 出力（エラー2件）→ ToolResult
Agent → read_file("Foo.cs")                → 内容取得
Agent → propose_edit("Foo.cs", 修正案)     [承認] → Editorで差分表示
User  → 承認
Agent → run_command("dotnet build")        [承認] → 再ビルド成功
Agent → 「修正完了。ビルドが通りました」    → AIバーに表示
```

人間が同じ作業をする時の経路（FolderTree→Terminal→Editor）と完全に一致しているのがポイント。

## 9. 状態管理・DI・ライフサイクル

- `App.xaml.cs` で `Host` を構築し、サービス／ViewModel／ツールをすべてDI登録。
- スコープ：サービスは概ね Singleton（Terminal/Editor/Workspace/AiClient/ToolRegistry）。ViewModelは用途に応じ Singleton/Transient。
- 会話状態は `ConversationStore`（Singleton）に集約。AIセッションの保存/復元に対応（ActivityBarの「AIセッション」から履歴閲覧）。

---

## 10. セキュリティ / 安全設計

- **コマンド実行とファイル書込は既定で承認制**（`ApprovalService`）。AIバーの承認カードで内容を確認してから実行。
- 「自動承認」モード（信頼ワークスペース向け）は設定でオプトイン。危険コマンドのブロックリスト/許可リストを用意。
- 実行スコープはワークスペースルート配下に限定（パストラバーサル防止）。
- APIキーは平文保存しない（Windows資格情報マネージャ / DPAPI）。
- すべてのツール実行はログに記録（監査・再現用）。

## 12. 決定事項 / 未決事項

### 決定済み（2026-05-31）
1. **.NET バージョン**：**.NET 9**（`net9.0` / WPFは `net9.0-windows`）。

## 13. 実装メモ（2026-05-31 初期実装完了）

実際にソリューションを生成し、ビルド成功（`slnexit=0`）・起動確認済み。実装過程で判明した
**sk0ya パッケージの実APIと設計への影響**を以下に記録する。

### 実パッケージ / 名前空間（要：当初想定と異なる）
| 項目 | 実際 |
|------|------|
| NuGet ID | `sk0ya.Terminal.Controls` / `sk0ya.Editor.Controls`（`.Core`/`.Defaults` を伴う） |
| ルート名前空間 | `Terminal.*` / `Editor.*`（`sk0ya.` 接頭辞は **無い**） |
| ターミナル埋め込みコントロール | **`Terminal.Tabs.TerminalTabView`** (UserControl)、ctor `(commandLine, workingDirectory)` |
| エディタ埋め込みコントロール | **`Editor.Controls.VimEditorControl`** (UserControl)。Vim操作・LSP・Git統合あり |

### 設計への影響（重要）
- **TerminalTabView は「対話型ターミナル」で、コマンドを実行して出力を取得する公開APIが無い**
  （`ExecuteCommand`+結果取得イベントが存在しない）。
  → エージェントの `run_command` は **独立した `Process`（PowerShell）をバックエンド**として実装。
    可視ターミナルは人間の対話用として併存。
  → **要相談**: TerminalTabView に「コマンド実行＋stdout/exit取得」APIを追加すれば、
    エージェント実行を可視ターミナルへ一本化できる（ユーザーが実装変更可能とのことなので有力）。
- **VimEditorControl** は `LoadFile` / `SetText` / `Text` / `FilePath` / `ExecuteCommand(exコマンド)` を公開。
  IEditorService はこれで実装。ただし **選択テキスト取得APIが無い**ため `get_selection` 相当は将来対応
  （`VimEngine` 経由 or API追加）。

### 現在の到達点（Phase 0〜5 相当が動作）
- 4ペイン+下部AIバー（展開式）レイアウト・ダークテーマ
- FolderTree（フォルダを開く→ツリー表示→ファイル選択でエディタに開く）
- TerminalTabView / VimEditorControl の埋め込み
- エージェントループ（StubAiClient で承認カード含むツール実行まで一通り動作）
- AIプロバイダ4種（Claude実装 / OpenAI・Local共通実装 / Copilotはスタブ）
- 既定プロバイダは **Stub**（APIキー不要で動作確認可能）

### 次の作業候補
1. 設定画面（プロバイダ切替・APIキー入力／資格情報マネージャ保管）
2. TerminalTabView へのコマンド実行API追加の可否決定（上記）
3. propose_edit の本格的な差分ビュー
4. セッション保存・復元
5. Copilot デバイス認証フロー
