# Loomo 設計書

> アプリ名: **Loomo**（ルート名前空間: sk0ya.Loomo）
> 作成日: 2026-05-31
> 最終更新: 2026-06-04

---

## 1. 概要

ローカルの開発ワークスペースを対象に動作する **AIエージェント搭載のデスクトップアプリ**（C# / WPF）。
ユーザーは自然言語で指示を出し、AIエージェントが **Terminal（コマンド実行）** と **Editor（ファイル表示・編集）** を操作しながらタスクを遂行する。

> **本アプリはローカルLLM専用**（2026-06-04 決定）。推論は端末上の **Ollama**（ネイティブ API `/api/chat`）で行い、
> クラウドAI（Claude / OpenAI / GitHub Copilot 等）には接続しない。設計の狙いはプライバシー・オフライン動作・
> 構成の単純さ。クラウドプロバイダを切り替える構想だった旧記述は §22 の決定に合わせて読み替えること。

### コアコンセプト
- AIエージェントは UI を「人間と同じ道具」として使う。すなわち FolderTree で対象を把握し、Terminal で実行し、Editor で表示・編集する。
- UI の各ペインは **疎結合なサービス + メッセージング** で連携し、AIエージェントは「ツール（Tool）」を通じてそれらを駆動する。
- モダンな見た目（ダーク基調・フラット・VS Code 系の操作感）と、機能を後から足しやすい拡張構造を両立する。

## 2. 技術スタック

| 領域 | 採用 | 理由 |
|------|------|------|
| ランタイム | .NET 9（`net9.0` / `net9.0-windows`） | 現行実装のターゲット |
| UIフレームワーク | WPF | 要件指定。リッチなデスクトップUIに最適 |
| アーキテクチャ | MVVM | テスト容易・View/ロジック分離 |
| MVVM基盤 | CommunityToolkit.Mvvm | `ObservableObject` / `RelayCommand` / `WeakReferenceMessenger` |
| DI / 起動 | Microsoft.Extensions.Hosting + DependencyInjection | サービス登録・ライフサイクル管理・設定・ロギングを統一 |
| ロギング | Microsoft.Extensions.Logging.Abstractions | Core 側は抽象に留める |
| Terminal | `sk0ya.Terminal.Controls` 1.0.4 | `TerminalTabView.RunCommandAsync` により可視ターミナル上で実行。既定シェルは `pwsh.exe` |
| Editor | `sk0ya.Editor.Controls` / `.Defaults` 1.0.1 | `VimEditorControl` と仮想ドキュメントを利用 |
| Browser | `Microsoft.Web.WebView2` | ブラウザペイン（Web タブ）。お気に入りアイコン取得・ナビゲーション |
| AIクライアント | ローカルLLM（Ollama ネイティブ API） | `IAiClient`＝`OllamaClient` 単一実装。`IAiClientFactory` 経由 |
| 設定保存 | `%APPDATA%/Loomo/settings.json` + DPAPI | APIキー（ローカルOllamaでは通常不要）を CurrentUser スコープで暗号化 |
| ワークスペース状態 | `%APPDATA%/Loomo/workspaces.json` | 複数ワークスペース・ペインレイアウト・各ペインのタブを保存 |

> AIプロバイダは `IAiClient` 抽象の背後に隠す。**実装はローカルの Ollama（`OllamaClient`）のみ**。
> 抽象は残してあるので、将来別のローカル推論バックエンドを足す余地はある（クラウド接続は対象外）。

## 3. 全体アーキテクチャ

レイヤー構成（依存は上→下の一方向）:

```
┌─────────────────────────────────────────────────────────┐
│  Presentation (WPF)                                       │
│   Views (XAML) ── ViewModels ── Behaviors / Converters   │
└───────────────┬─────────────────────────────────────────┘
                │  Messenger（イベント） / DI
┌───────────────▼─────────────────────────────────────────┐
│  Application / Agent Core                                 │
│   AgentOrchestrator（エージェントループ）                 │
│   ToolRegistry / IAgentTool 群                            │
│   ApprovalService（実行承認）/ ConversationStore          │
└───────────────┬─────────────────────────────────────────┘
                │  サービス抽象（インターフェース）
┌───────────────▼─────────────────────────────────────────┐
│  Services / Adapters                                      │
│   ITerminalService → sk0ya.Terminal                      │
│   IEditorService   → sk0ya.Editor                        │
│   IWorkspaceService→ ファイルシステム / FolderTree状態     │
│   IAiClient        → OllamaClient（ローカルLLM / Ollama）  │
└──────────────────────────────────────────────────────────┘
```

### 連携の要点
- **View 同士は直接参照しない。** すべて `WeakReferenceMessenger` のメッセージ、または共有サービス経由でやり取りする。
- AIエージェントは UI を直接触らず、**ツール → サービス → （メッセージ）→ View** という経路で操作する。これにより「AIが操作する」「人間が操作する」が同じ経路に集約され、テスト・記録・再現が容易になる。

## 4. UIレイアウト（現行：ツリー構造ペイン）

採用: **カスタムタイトルバー + ActivityBar/Sidebar + 自由配置のペイン群（AI もペインの1つ）**

> 旧案（全幅下部AIバー）から刷新。メイン領域は固定 Grid ではなく **レイアウトツリー**で表現し、
> ユーザーがペインをドラッグで 2D 並べ替え・分割・リサイズ・表示切替できる（§21）。AI は専用バーではなく
> 1つのペインになった。

```
┌──────────────────────────────────────────────────────────────┐
│ 🤖 Loomo  [ワークスペース▼] [＋]            [─][□][×]        │ ← カスタムタイトルバー
├────┬─────────────────────────────────────────────────────────┤
│ 🗂 │            │  EDITOR ＋ ─  │  BROWSER  ◀▶⟳[url] Go ＋ ─ │
│ 📑 │  Folder    │ ┌──tabs────┐ │ ┌──tabs──┐                  │
│ 💬 │  Tree /    │ │ a.cs │b.cs│ │ │ tab1   │                  │
│ ── │  Tabs /    │ └──────────┘ │ └────────┘                  │
│ ▦  │  Sessions /│  VimEditor    │   WebView2                  │
│ ✎  │  Settings  ├───────────────┴─────────────────────────────┤
│ 🌐 │            │  TERMINAL ＋ ─   [tab1][tab2]                │
│ 🤖 │  (排他切替) │  TerminalTabView                            │
│ ── │            ├─────────────────────────────────────────────┤
│🎨⚙ │            │  AI ─   会話履歴 / ツールカード / 入力欄     │
└────┴────────────┴─────────────────────────────────────────────┘
```

### 各領域
| 領域 | 内容 | サイズ |
|------|------|--------|
| タイトルバー | `WindowChrome` のカスタムバー（`WindowStyle=None`）。アプリ名・ワークスペース選択 ComboBox・フォルダ追加・最小化/最大化/閉じる。最大化時はタスクバーを覆わない（`WM_GETMINMAXINFO`） | 高さ36px |
| ActivityBar | アイコン縦並び。上＝Sidebar 切替（🗂エクスプローラ/📑タブ/💬AIセッション）、区切り線の下＝ペイン表示トグル（▦Terminal/✎Editor/🌐Browser/🤖AI）、最下部（下寄せ）＝🎨外観・⚙設定。ウィンドウ下端まで伸びる | 固定 48px |
| Sidebar | Explorer / Tabs / Sessions / Settings / Appearance の5パネルを排他表示（`ShellViewModel.ActivePanel`＋`EnumToVisibilityConverter`）。同じアイコン再クリックで閉じる | 可変・GridSplitter（閉時は幅0） |
| FolderTree | `.gitignore` 非表示、変更ファイルのみ表示、FileSystemWatcher 自動更新。単クリック選択／ダブルクリックで新エディタタブ。Vim 風キーボード操作（j/k 上下・h 折りたたみ/親・l/Enter 展開/開く・gg/G 先頭/末尾）と `/` インクリメンタル検索（入力中に逐次ヒット、n/N で巡回、Enter 確定 / Esc で元の選択へ） | Sidebar 内 |
| Tabs パネル | Terminal/Editor/Browser の全タブを横断一覧。クリックで該当タブをアクティブ化・×で閉じる | Sidebar 内 |
| Terminal ペイン | `TerminalTabView`（既定 `pwsh.exe`）をホスト。複数タブ。AI実行コマンドも同じ可視ターミナルに流す | ツリー内の可変ペイン |
| Editor ペイン | `VimEditorControl` をホスト。複数タブ・通常ファイル・仮想ドキュメント | ツリー内の可変ペイン |
| Browser ペイン | `WebView2`。複数タブ・アドレスバー・戻る/進む/更新・favicon | ツリー内の可変ペイン |
| AI ペイン | `AiBarView`（会話履歴・ツール実行カード・入力欄）を1ペインとして配置 | ツリー内の可変ペイン |

### レイアウト実装方針
- メイン領域は **レイアウトツリー**（`PaneNode` = リーフ `PaneLeaf`／スプリット `PaneSplit`）で保持し、`PaneHost`（Grid）を
  `RebuildPaneLayout` がツリーから動的に組み立てる。スプリットごとに行/列 Grid を生成し、ペイン本体を再ペアレントする
  （同一ウィンドウ内なので WebView2 等は生存）。詳細は §21。
- 各ペインは **ドラッグで 2D 並べ替え**（ドロップ先の辺を半透明プレビューで表示。WebView2 の airspace を越えるため
  最前面 `Popup` で描画）、**GridSplitter でリサイズ**、**タイトルの「—」で非表示**（ActivityBar のトグルで再表示）。
- 既定レイアウトは `[Editor | Browser]`（上段・横並び）/ `[Terminal]` / `[AI]`（縦積み）。
- サイドバー切替は `ShellViewModel.ActivePanel`、開閉は `IsSidebarVisible`（列幅と GridSplitter を code-behind で切替）。
- 配色は `Themes/Palette.<Theme>.xaml`（ブラシのみ・Dark / Light / SolarizedDark / Nord / HighContrast）に分離し、
  角丸・余白・スタイル類は `Themes/Controls.xaml` に集約する。スタイルは色を `DynamicResource` で参照するため、
  `ThemeManager` がマージ辞書のパレットを差し替えるだけで全 UI に即時反映される。
- アクセントカラーは `ThemeManager` が `Application.Resources` 直下に `Accent` / `AccentHover` / `AccentFg` を置いて
  パレット定義を上書きする（マージ辞書より直下キーが優先。`AccentHover` は自動で明色化、`AccentFg` は輝度から白/黒を選択）。
  空ならテーマ既定へ戻る。
- テーマ／アクセントの選択は **外観（Appearance）パネル**（ActivityBar の 🎨）で行い、AIプロバイダ設定（⚙）とは分離。
  どちらも `settings.json` に永続化される。

## 6. 主要コンポーネント設計

### 6.1 サービス抽象（Abstractions）

```csharp
public interface ITerminalService
{
    Task<CommandResult> RunCommandAsync(string command, CancellationToken ct);
    void SetWorkingDirectory(string path);
    string CurrentDirectory { get; }
    bool IsExecuting { get; }
    event EventHandler<CommandResult>? CommandExecuted;
}

public interface IEditorService
{
    Task OpenFileAsync(string path);
    Task<string> GetActiveContentAsync();
    Task<string> GetSelectedTextAsync();                       // 現状は Editor API 制約により空文字
    Task<string> ShowDiffAsync(string path, string proposedContent);
    Task<bool> ApplyEditAsync(string path, string newContent); // 適用（要承認）
    Task OpenDocumentAsync(EditorDocument document);           // 設定編集用の仮想ドキュメント
    string? ActiveFilePath { get; }
}

public interface IWorkspaceService
{
    string? RootPath { get; }
    string? SelectedPath { get; set; }       // FolderTreeの選択
    void OpenFolder(string rootPath);
    Task<IReadOnlyList<FileNode>> ListAsync(string path);
    Task<string> ReadFileAsync(string path);
    string ResolvePath(string path);         // ルート配下制限込みのパス解決
    event EventHandler<string?>? SelectionChanged;
    event EventHandler<string?>? RootChanged;
}
```

これらの **実装** は `Loomo.Services` に置き、sk0ya.Terminal / sk0ya.Editor のコントロールをラップする。
（sk0ya.* の実APIに合わせてアダプタ内で吸収。API差異の影響をこの層に閉じ込める。）

### 6.2 AIクライアント

```csharp
public interface IAiClient
{
    IAsyncEnumerable<AgentEvent> StreamAsync(
        Conversation conversation,
        IReadOnlyList<ToolDefinition> tools,
        CancellationToken ct);
}
// AgentEvent = TextDelta | ThinkingDelta | ToolUseRequested | TurnCompleted ...
```

ストリーミングで `TextDelta`（逐次テキスト）・`ThinkingDelta`（思考）・`ToolUseRequested`（ツール呼び出し要求）を流す。

**実装は `OllamaClient` のみ**（プロバイダ `Local`）。ローカルの **Ollama ネイティブ API**（`POST /api/chat`）を
NDJSON ストリーミングで叩く（`OllamaProtocol`）。思考は `message.thinking` として本文と分離して届き、`think`
（真偽値）で有効・無効を制御する（設定 `Thinking`（bool）→ `think`。Ollama の think は真偽値のため段階指定は持たない）。
モデル別の最適化は `ModelProfiles`（tools/thinking の対応可否・`num_ctx`・サンプリングをモデル名から解決）が担う。
tool_calls の `arguments` はオブジェクトで授受。モデル一覧・起動確認は `/api/tags`。ホスト未起動なら
`OllamaLauncher` が `ollama serve` を試行する。クラウドAPI（Anthropic / OpenAI / Copilot）には接続しない。

## 7. AI連携設計（最重要）

### 7.1 ツール（Tool）= AIの手足

AIエージェントは各プロバイダの function calling / tool use 形式を通じて以下のツールを呼び出す。各ツールはサービスを叩くだけの薄いラッパ。

| ツール名 | 接続先サービス | 説明 | 承認 |
|----------|----------------|------|------|
| `list_directory` | IWorkspaceService | フォルダ内容の取得 | 不要 |
| `read_file` | IEditorService/FS | ファイル読み取り | 不要 |
| `open_in_editor` | IEditorService | Editorで開く | 不要 |
| `propose_edit` | IEditorService | 差分を提示（適用は承認後） | **必要** |
| `run_command` | ITerminalService | Terminalでコマンド実行 | **必要** |
| `get_selection` | IWorkspaceService | FolderTreeの現在選択を取得 | 不要 |

```csharp
public interface IAgentTool
{
    string Name { get; }
    ToolDefinition Definition { get; }              // JSON Schema（LLMへ渡す）
    bool RequiresApproval { get; }
    Task<ToolResult> ExecuteAsync(JsonElement args, CancellationToken ct);
}
```

新しい能力の追加 = `IAgentTool` を実装して DI 登録するだけ（**拡張ポイント**）。

### 7.2 エージェントループ（AgentOrchestrator）

```
ユーザー入力（下部AIバー）
        │
        ▼
[1] Conversation に user メッセージ追加
        │
        ▼
[2] IAiClient.StreamAsync(会話, ツール定義)
        │
        ├── TextDelta ───────────► AIバーに逐次表示
        │
        └── ToolUseRequested
                │
                ▼
        [3] SafetyPolicy.Evaluate
                ├─ Blocked → エラー結果をAIへ返す
                └─ Allowed
                ▼
        [4] RequiresApproval && !AutoApprove?
                ├─ Yes → ApprovalService → AIバーに承認カード表示 → ユーザー判断
                └─ No  → そのまま実行
                │
                ▼
        [5] ToolRegistry.Get(name).ExecuteAsync(args)
                │  （= Terminal実行 / Editor操作 / FS読取）
                ▼
        [6] ToolResult を Conversation に追加 → [2] へ戻る
        │
        ▼
最終テキストで完了（TurnCompleted）
```

- ループ・状態は `Loomo.Core` 側に持たせ、UIはイベント購読のみ。
- 中断（キャンセル）対応：`CancellationToken` を全層に伝播。

### 7.3 シーケンス例：「このプロジェクトをビルドしてエラーがあれば直して」

```
User → AIバー：「ビルドしてエラーを直して」
Agent → run_command("dotnet build")        [承認] → Terminalで実行
Terminal → 出力（エラー2件）→ ToolResult
Agent → read_file("Foo.cs")                → 内容取得
Agent → propose_edit("Foo.cs", 修正案)     [承認] → Editorで差分表示
User  → 承認
Agent → run_command("dotnet build")        [承認] → 再ビルド成功
Agent → 「修正完了。ビルドが通りました」    → AIバーに表示
```

人間が同じ作業をする時の経路（FolderTree→Terminal→Editor）と完全に一致しているのがポイント。

## 8. 拡張性の設計

| 拡張軸 | 仕組み |
|--------|--------|
| サイドバーパネル追加 | 現状は `ShellViewModel.ActivePanel` による固定5パネル（Explorer/Tabs/Sessions/Settings/Appearance）。`IPanelContribution` 方式は将来の拡張案 |
| ペイン配置 | レイアウトツリー（`PaneNode`）でドラッグ並べ替え・分割・リサイズ・表示切替に対応（§21）。新ペイン種別は `PaneKind` 追加＋ホスト要素追加で拡張 |
| AIツール追加 | `IAgentTool` 実装 + DI登録（ToolRegistryが自動収集） |
| AI推論バックエンド | `IAiClient` の別実装に差替（**ローカル推論のみ対象**。現状は Ollama の `OllamaClient`） |
| テーマ | パレット（`Palette.*.xaml`）を分離し `ThemeManager` で実行時切替。Dark/Light/SolarizedDark/Nord/HighContrast の5種＋アクセントカラー上書き。新テーマは `AppTheme` 追加＋同名パレット追加＋外観パネルに1行追加で拡張 |
| コマンド/ショートカット | 将来 `ICommandContribution` でコマンドパレット・キーバインド登録 |
| 設定 | `%APPDATA%/Loomo/settings.json`。APIキーは DPAPI で暗号化 |

> 「ドッキング自由化」は AvalonDock を導入せず、独自のレイアウトツリー＋ `PaneHost` 動的構築で実装した（§21）。
> WebView2 を含むペインを再ペアレントしても生存させられる軽量な方式を選んだ。

## 9. 状態管理・DI・ライフサイクル

- `App.xaml.cs` で `Host` を構築し、サービス／ViewModel／ツールをすべてDI登録。
- スコープ：サービスは概ね Singleton（Terminal/Editor/Workspace/AiClient/ToolRegistry）。ViewModelは用途に応じ Singleton/Transient。
- 会話状態は `ConversationStore`（Singleton）に集約。AIセッションの保存/復元に対応（ActivityBarの「AIセッション」から履歴閲覧）。

## 10. セキュリティ / 安全設計

- **コマンド実行とファイル書込は既定で承認制**（`ApprovalService`）。AIバーの承認カードで内容を確認してから実行。
- 「自動承認」モード（信頼ワークスペース向け）は設定でオプトイン。危険コマンドのブロックリスト/許可リストを用意。

### 19.1 エージェントの観測性（Observability）

- **動作中エージェントの可視化**: 実行中に「いまどのエージェントが動いているか」を UI で分かるようにする
  （後述のマルチエージェントと連動）。
- **プレイバック**: 保存したログ／セッションを再生（タイムライン的に追体験）できるようにする。
  ※ §3 の「AIも人間も同じ経路で操作する」設計が、記録・再現を容易にする前提として効いてくる。

## 20. 観測性＋プレイバック 詳細設計（2026-06-01 構想深掘り）

> §19.1 / §19.1（プレイバック）の深掘り。**未確定の設計案**だが、既存コード（`AgentOrchestrator` /
> `AgentEvent` / `ConversationStore`）への接続点まで具体化する。マルチエージェント可視化（§19.2）の土台でもある。

### 20.1 中心となる気づき：「最終状態」ではなく「イベントストリーム」を保存する
- 現状 `ConversationStore`（§16）が保存するのは `Conversation.Messages`＝**最終状態のスナップショット**のみ。
  これは「再開（resume）」には十分だが、観測・解析・再生には **以下を取りこぼしている**：
  - 各処理のタイミング（time-to-first-token / ツール所要時間 / ターン全体の所要）
  - 使用プロバイダ・モデル（ターン毎に `AiClientFactory.ResolveCurrent()` で変わり得る）
  - 安全ポリシーのブロック有無と理由（`SafetyPolicy.Evaluate` の結果）
  - 承認の可否と待ち時間（`ApprovalRequested` → 人間の判断）
  - ループ反復回数（`MaxIterations=25` のどこで終わったか）
  - エラー（`AgentError`）、中断（`OperationCanceledException`）
  - トークン/コスト（現状クライアントが表に出していない＝既知の制約）
- 一方 `AgentOrchestrator.RunTurnAsync` は **すでに豊富な `AgentEvent` ストリームを流している**
  （`TextDelta` / `ToolUseRequested` / `ToolExecutionStarted` / `ToolExecutionCompleted` /
  `ApprovalRequested` / `TurnCompleted` / `AgentError`）。
  → **観測性の核心 = この順序付きイベント列を追記専用ログとして永続化すること。**
  セッションJSON（再開用の状態）とは **別ライフサイクル・別ファイル**として分離する。

### 20.2 ストレージ：セッション毎の追記専用 JSON Lines
- 形式: **JSONL**（1行1イベント）。`%APPDATA%/Loomo/traces/{sessionId}.jsonl`。
  - 追記専用 → クラッシュ耐性・ストリーム書き込み・順序保証・末尾 tail で「実況」可能。
  - `sessions/{id}.json`（§16）は従来どおり **再開用スナップショット**として残す（役割分担）。
- 大きなペイロード（ツール出力・ファイル内容）は、小さければインライン、大きければ
  `traces/{sessionId}/blobs/{sha256}` にスピルしてハッシュ参照（JSONL を走査可能なサイズに保つ）。

### 20.3 トレースイベントのスキーマ（封筒＋種別）
```
封筒(envelope): { ts, seq, sessionId, turnId, agentId, kind, payload }
   agentId は単一エージェント期は "root" 固定。§19.2 で複数エージェントが同一ログに interleave。
```
種別（kind）と payload 例：
| kind | payload | 取得元 |
|------|---------|--------|
| `session.started` | workspaceRoot, provider, model, settingsSnapshot | ターン開始時（新規セッション） |
| `turn.started` | turnId, userInput, provider, model | `RunTurnAsync` 冒頭 |
| `ai.message` | fullText（TextDelta を合算） | ストリーム終了時 |
| `ai.tool_use` | toolUseId, name, argsJson | `ToolUseRequested` |
| `safety.evaluated` | tool, blocked, reason | `SafetyPolicy.Evaluate` 後 |
| `approval.requested` | tool, summary | `ApprovalRequested` |
| `approval.resolved` | tool, approved, waitMs | 承認完了時 |
| `tool.started` | toolUseId, name | `ToolExecutionStarted` |
| `tool.completed` | toolUseId, name, isError, durationMs, contentLen, contentRef | `ToolExecutionCompleted` |
| `ai.usage` | inputTokens, outputTokens, costEstimate | クライアント（要拡張・§20.7） |
| `turn.completed` | finalText, iterations, durationMs | `TurnCompleted` |
| `error` | message, where | `AgentError` / 例外 |

### 20.4 フック方法（既存への侵襲を最小化）
- **`ITraceSink` を `AgentOrchestrator` にDI注入**する案を推奨。
  - `ITraceSink.Record(TraceEvent)`（fire-and-forget、内部でバッファ＋非同期フラッシュ）。
  - Core を汚さないよう **`NullTraceSink.Instance`（no-op）を既定**にし、既存テストはそのまま通す。
  - UI 向けの `AgentEvent` ストリームは**一切変えない**（`AiBarViewModel` は無改修）。
- 単純な「`RunTurnAsync` の yield をデコレータで tee する」案だけでは、`AgentEvent` に乗っていない
  情報（プロバイダ/モデル・安全ブロック理由・承認待ち時間・各種 duration・トークン）が取れない。
  → orchestrator 内部の既知ポイントで `_trace.Record(...)` を呼ぶ方式が確実。
- 実 IO 実装 `JsonlTraceSink` は **Loomo.Core に置いて可**（`ConversationStore` が既に Core で File IO を
  しており一貫する）。DI で `ITraceSink` をシングルトン登録（`App.xaml.cs`）。

### 20.5 解析層
- `TraceReader`：JSONL を読み、型付き `TraceEvent` を列挙。
- **セッション単位メトリクス**（`SessionMetrics`）：ツール呼び出し回数・成功/失敗率・総所要・
  平均ツール所要・ターン毎反復数・time-to-first-token・（取れれば）総トークン/コスト。
  → Sessions パネルに「セッション分析」タブとして表示。
- **横断分析**（後続）：失敗の多いツール、期間別トークン消費など。JSONL 全走査が重くなったら
  SQLite インデックスを導入（まずはオンデマンド走査で開始）。

### 20.6 プレイバック
- §3 の不変条件（AIも人間も同じ経路で操作）＋ トレースが順序付きイベント列であることにより、
  プレイバック = **トレースをタイムライン上に再生して読み取り専用トランスクリプトを再構成**する。
  - **モードA：トランスクリプト再生（安全・v1）**。`TranscriptEntry` 描画を再利用し、
    元のタイミング（または step 実行）で会話を再構築。**ツールは再実行しない**
    （`run_command` の再実行は破壊的になり得るため）。
  - **モードB：再実行/フォーク（高度・危険・将来）**。ユーザー入力を実エージェントに再投入して
    再現する。明示オプトイン＋サンドボックス前提。v1 はスコープ外として記載のみ。
- UI：Sessions パネル内に Playback ビュー（スクラバー/タイムライン・再生/一時停止/ステップ・速度）。

### 20.7 既知制約・横断事項
- **トークン/コスト**：現状クライアントは usage を表に出していない（§AI clients の既知制約）。
  `ai.usage` を埋めるにはクライアント側の応答パースに usage 抽出を足す必要がある（別フェーズ）。
- **プライバシー/サイズ**：トレースはプロンプト・ファイル内容・コマンド出力を含む機微データ。
  既定オプトアウト可、保持ポリシー（件数/日数上限でローテーション）、サイズ上限を設ける。
  API キーは元々 payload に載らない。
- **マルチエージェント前方互換**：封筒に `agentId` を最初から持たせれば、§19.2 到達時に同一ログへ
  自然に interleave でき、§19.1「動作中エージェントの可視化」は **トレース末尾のライブ tail を
  agentId/状態でフィルタするだけ**で実現できる。観測性が §19.2 の土台になる所以。

## 21. ペインレイアウト刷新・ブラウザ・複数ワークスペース（2026-06-01 実装完了）

> §4 を全面刷新した実装の記録。「全幅下部 AIバー＋固定 Grid」を廃し、**メイン領域をレイアウトツリーで保持して
> 動的に組み立てる**方式へ移行。あわせて Browser ペイン・各ペインのタブ・複数ワークスペースを導入した。
> 中核ロジックは `ShellWindow.xaml.cs`（code-behind）に集約している。

### 21.1 レイアウトツリーと動的構築
- ツリーは `PaneNode`（基底）＝ `PaneLeaf`（1ペイン）／ `PaneSplit`（行=上下 or 列=左右の入れ子）で表す。
  ルートを `_root` に保持し、`RebuildPaneLayout` が **スプリットごとに Grid を生成**して `PaneHost` を組み直す。
  ペイン本体（`TerminalPane`/`EditorPane`/`BrowserPane`/`AiPane`）は XAML に1つだけ定義し、再構築のたびに生成した
  Grid へ **再ペアレント**する（同一ウィンドウ内なので WebView2/Terminal の HWND は生存）。
- **正規化** `Normalize`：空スプリット除去・子1個のスプリット畳み込み・同方向スプリットのフラット化。
- **既定レイアウト** `ApplyDefaultLayout`：`[Editor | Browser]`（列）/ `[Terminal]` / `[AI]`（行）。
- 列/行トラックには可視な子だけを割り当て、間に `GridSplitter` を差す。`GridSplitter` 操作後の star 比率は
  `CaptureLayoutSizes`→`CaptureNode` で各ノードの `Weight` に取り込み、保存・再構築で維持する。

### 21.2 ドラッグ並べ替え（スナップ風プレビュー）
- ペインタイトルを掴んでしきい値以上ドラッグすると `BeginPaneDrag`。**WebView2 の airspace を越えて最前面に描く
  ため、透明 `Popup`＋`Canvas`** を被せ、その上でマウス追跡＆プレビュー（ドロップ先ペインの枠＋落とす辺の半分を
  半透明で塗る）を行う。
- `HitTestCell` でカーソル下のペイン矩形を求め、`NearestZone` で最も近い辺（Left/Right/Above/Below）を判定。
  ドロップで `MovePane`→`RemoveNode`＋`InsertRelative`：望む方向が対象の親スプリットと一致すれば兄弟挿入、
  異なれば対象を新スプリットで包む（＝列の片方だけを上下分割できる）。

### 21.3 ペインの表示切替
- タイトルの「—」(`OnHidePane`) でそのペインを隠す。ActivityBar のトグル（▦/✎/🌐/🤖, `OnTogglePaneVisibility`）で
  再表示。非表示はツリーから消さず `PaneLeaf.Hidden` を立てるだけなので、**再表示で元の位置・比率に戻る**。
  可視ペインが1枚のときは最後の1枚を隠さない。
- **ペイン間フォーカス移動（Ctrl+W → h/j/k/l, vim 風）**: `Ctrl+W` でプレフィックス待ち状態に入り、続く
  `h`/`j`/`k`/`l` でフォーカス中ペインの左/下/上/右の隣接ペインへフォーカスを移す（`FocusPaneInDirection`）。
  方向は `HitTestCell`/`NearestZone` と同じ矩形判定でフォーカス中ペインに最も近い隣接ペインを選ぶ。

### 21.4 タブ（Terminal / Editor / Browser）
- 各ペインは複数タブを持つ。本体（`TerminalTabView`/`VimEditorControl`/`WebView2`）は専用の ContentHost に重ね、
  アクティブなものだけ `Visible` にする。タブ見出しは各ペイン上部のタブストリップ（横スクロール対応）に表示。
- サイドバーの **Tabs パネル**（`TabsViewModel`）が3種のタブを横断一覧化。クリックで該当タブをアクティブ化、×で閉じる
  （`TabActivated`/`TabCloseRequested` を code-behind が購読）。タブのアイコンは `TabIconService` が供給：
  ファイルは Shell アイコン（`SHGetFileInfo`）、ターミナルは pwsh/powershell の実行ファイルアイコン、
  ブラウザは WebView2 の favicon（`GetFaviconAsync`、取得失敗時は Edge アイコン→ベクターのフォールバック）。

### 21.5 Browser ペイン（WebView2）
- `Microsoft.Web.WebView2` を新規ペインとして追加。アドレスバー（URL/検索クエリを `NormalizeBrowserAddress` で解釈、
  localhost は http、空白を含めば Google 検索）・戻る/進む/更新・favicon・複数タブに対応。既定 URL は Google。
- `NavigationCompleted`/`FaviconChanged` でタブのタイトル・アイコン・アドレス表示を更新。

### 21.6 複数ワークスペースと永続化
- タイトルバーの ComboBox（`WorkspaceListViewModel`）でワークスペースを切替、「＋」で `OpenFolderDialog` から追加。
- 状態は `WorkspaceStateStore`（`%APPDATA%/Loomo/workspaces.json`）に保存。`WorkspaceSnapshot` が
  RootPath・各ペインのタブ（`TerminalTabSnapshot`/`EditorTabSnapshot`/`BrowserTabSnapshot`）・アクティブタブ・
  ペインレイアウト（`PaneNodeSnapshot`）を保持。`SaveActiveWorkspaceSnapshot` がタブ操作・レイアウト変更・終了時に
  `CaptureInto` でスナップショットを更新して書き出す（即時永続化）。
- ワークスペース切替 `SwitchWorkspaceAsync` は現ペインのタブを Detach→対象の保存済みタブ/レイアウトを復元。
  各ワークスペースのライブなタブ集合はメモリ上（`_terminalWorkspaces` 等）にもキャッシュし、再切替で素早く Attach する。
- フォルダを開くと FolderTree のルート・エージェントの作業ディレクトリ（`TerminalService.SetWorkingDirectory`）も同期。

### 21.7 カスタムタイトルバー
- `WindowStyle=None` + `WindowChrome`。最小化/最大化(復元)/閉じるを自前ボタンで実装。`WindowStyle=None` のままだと

## 22. ローカルLLM専用化 ＋ Ollama ネイティブAPI移行（2026-06-04 実装完了）

> **決定**: Loomo は **ローカルLLM（Ollama）専用**とする。クラウドAI（Claude / OpenAI / GitHub Copilot）への
> 接続は廃止。狙いはプライバシー・オフライン動作・構成の単純さ。本節が AIプロバイダに関する正本で、
> §2/§3/§6.2/§12 の旧マルチプロバイダ記述はこれに読み替える。

### 22.1 背景（なぜネイティブAPIへ）
- 旧実装は Ollama の **OpenAI互換エンドポイント**（`/v1/chat/completions`・SSE）を使い、思考量を
  `reasoning_effort`（none/low/medium/high）で送っていた。しかし **qwen3 等は `reasoning_effort` を解釈せず**、
  `none` を指定しても思考（`<think>…</think>`）が出力され続ける問題があった（`reasoning_effort` は実質 gpt-oss 流儀）。
- Ollama **ネイティブ API**（`/api/chat`）は思考の有効・無効を真偽値 `think` で確実に制御でき、思考も
  `message.thinking` として本文と分離して返す。これに全面移行して問題を解消した。

### 22.2 変更点
- **削除**: `OpenAiProtocol`（OpenAI互換ワイヤ処理）・`OpenAiCompatibleClient`、および設計上存在した
  クラウド系クライアント（Claude / Copilot / Stub）と Copilot 認証（§17）。
- **新設**: `Loomo.Ai/Clients/OllamaProtocol`（`/api/chat` のリクエスト組立＋NDJSON ストリーミング）と
  `OllamaClient`（プロバイダ `Local`）。`AiClientFactory` は `OllamaClient` を返す。
- **思考制御**: `ProviderConfig.Thinking`（bool）をそのまま `think` へ写像。thinking 対応モデルのみ送出する
  （`ModelProfiles` が判定。非対応モデルは送らない＝`think:true` でのエラーを回避）。旧 `ThinkingEffort`
  （none/low/medium/high）は `AiSettingsStore` が読み込み時に bool へ移行する。
- **イベント化**: 思考は `message.thinking`→`ThinkingDelta`、本文は `message.content`→`TextDelta`、
  ツールは `message.tool_calls`（`arguments` は**オブジェクト**）→`ToolUseRequested`。`done:true` で完了。
  旧来の `<think>` タグ分離パーサ（`ThinkStreamParser`）は不要となり廃止。
- **モデル一覧・起動確認**: `/v1/models` → **`/api/tags`**（`models[].name`）。`OllamaLauncher` の死活確認も `/api/tags`。
- **ホスト正規化と後方互換**: 既定ホストを `http://localhost:11434/v1` → **`http://localhost:11434`**。
  `OllamaLauncher.ResolveHost` が設定 `BaseUrl` 末尾の `/v1` を自動除去するため、旧 `settings.json` もそのまま動く。
- **ツール非対応モデル**: ネイティブAPIが「does not support tools」を返したら、tools 無しで1回だけ再送する挙動は維持。
