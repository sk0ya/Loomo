# ローカルLLMをエージェントで実用化する

## Loomo 開発で得た実測ベースの知見

<br>

C# / WPF (.NET 9) デスクトップAIエージェント
ローカル推論・CPU実行・小型 int4 モデル

<br>

<span class="small">出典: `docs/エージェントループ知見.md` / 評価ハーネス実測 / メモリ</span>

---

## Loomo とは

- **ローカルで動く AI コーディングエージェント**（C# / WPF / .NET 9）
- 自然言語の指示を **ツール呼び出し（function calling）** で実行
  - `run_powershell` / `write_file` / `edit_file` / `web_search`
- AI が **Terminal / Editor / FolderTree** の見えるペインを操作
- **HTTP もサーバも無し** — 推論はすべて **in-process**
- 前提環境は **Windows + CPU + 小型 int4 モデル**（API キー不要）

> 「クラウドの巨大モデル」ではなく「手元の non-thinking 小型モデル」で
> どこまでエージェントを成立させられるか、が一貫したテーマ

---

## 大前提：このマシンの制約

| 要素 | スペック |
|---|---|
| CPU | Ryzen 5 3500（6コア） |
| GPU | GeForce GT 710 / VRAM 1GB（2014年・実質無力） |
| RAM | 16GB |

- **GPU が非力 → モデルが載らず実質 CPU 実行**
- VRAM 1GB に 2〜4GB のモデルは載らない
- RAM 16GB が **量子化を上げる方向を塞ぐ**天井（fp32 は重みだけ ~15GB）
- → **int4 は攻めた選択ではなく、事実上の唯一解**

📌 すべての知見は「**非力GPU + CPU実行 + int4**」という制約の上にある

---

## 知見の全体像

1. **パフォーマンス** — 遅延の正体は何か
2. **バックエンドの変遷** — Ollama → ONNX → llama.cpp
3. **小型モデルの失敗と対策** — 能力はコードに、プロンプトは最小限
4. **評価** — 実機ハーネスと過適合の罠
5. **ツール設計** — なぜツールは少数なのか

---

# 1. パフォーマンス

## 遅延の正体は **prefill**（プロンプト評価）

---

## 遅延の主因は decode でも load でもない

実測（phi4-mini 3.8B / num_ctx 8192 / CPU）:

| 状況 | 入力tok | prefill | decode | load |
|---|--:|--:|--:|--:|
| 「pwd」turn1 | 471 | **18.9s** | 1.4s | 0.28s |
| 連続 turn1 | 383 | **18.7s** | 9.3s | 0.29s |
| 連続 turn2 | 486 | **36.5s** | 10.2s | 0.29s |

- **遅さの主因は prefill**。CPU で prefill ~13–20 tok/s、decode ~10 tok/s
- `load` は ~0.3s ＝ 重みは常駐済み（cold cost は初回だけ）
- prefill 時間は**トークン数に綺麗に比例しない** → cold/warm と会話長が支配的

---

## なぜ遅いか＝GPUが非力でCPUに落ちている

- `ollama ps` → `PROCESSOR 97%/3% CPU/GPU`
- GT 710 に **3%だけオフロード** → 古い Kepler GPU と細い PCIe が
  かえって **prefill のボトルネック**に
  - Ollama 0.30.5 で prefill **200→6 tok/s（35倍悪化）**、"pwd" 1ターン 99秒
- **`num_gpu=0`（100% CPU）で 6→39〜56 tok/s に回復**（end-to-end 8倍）

> 教訓：中途半端なGPUオフロードは CPU 単独より遅い。
> 弱いGPUなら **完全に切る**のが正解。

---

## 反復回数 × prefill が支配的

- 1イテレーション = AI 1呼び出し = **prefill 1回**
- ツール呼び出しが**失敗して往復が増えると、その回数ぶん prefill を払い直す**
- 実測（qwen3:1.7b）: pwsh が2回エラー → AI 3往復で **合計 1m15s**

> 💡 **「速いが間違える小型モデル」より「やや遅いが一発で当てるモデル」が
> 総時間で勝つことがある。**

- モデル選定は decode tok/s だけでなく **tool call の一発成功率**で評価
- 例：qwen3:1.7b は「pwd」に3回連続で的外れ／phi4-mini は一発で `Get-Location`

---

## 効く対策（効果順）

1. **モデルを小さくする**（無料・即効・最有力）
   - CPU速度はパラメータ数にほぼ**反比例**（3.8B→1.7B で約2倍）
   - トレードオフは tool call 安定性 → やみくもには下げられない
2. **GPU増設**（VRAM 8GB以上）— ハード投資
3. **コンテキストを切り詰める** — 会話が伸びると線形以上に悪化（turn2=36.5s）
   - `num_ctx` は意図的に 8192 に固定

> トークン削減（プロンプト圧縮）より **モデルサイズ削減のほうが桁で効く**

---

# 2. バックエンドの変遷

## Ollama → ONNX Runtime GenAI → llama.cpp

---

## Ollama HTTP から ONNX in-process へ全面移行

- **2026-06: Ollama を完全撤去**し、**ONNX Runtime GenAI（in-process / CPU）** へ
- パッケージ `Microsoft.ML.OnnxRuntimeGenAI`、**HTTP もサーバも無し**
- `OnnxGenAiEngine` が Model/Tokenizer を**常駐**（cold cost を一度だけ払う）
- 生成は batch-1・SemaphoreSlim で直列化・別スレッド → `Channel<AgentEvent>`

**移行で見えた cold cost の真因**

- 「1回目だけ十数秒」は decode/prefill ではなく **重みのページイン**
- `new Model()` は mmap した ~2GB を全ページ触らない → 初回 prefill でディスク→RAM
- → **起動時にダミー生成1回（WarmUp）**で全レイヤーをページイン

---

## KVキャッシュ：Ollamaでは「効かない」→ ONNXで「効く」

**Ollama 時代（§2）**

- turn2(36.5s) が turn1(18.7s) より遅い ＝ **毎ターン全文を再prefill**
- プレフィックスKVキャッシュは当てにならない

**ONNX 移行後（§2.1）— 状況が変わった**

- 常駐 `Generator` ＋ `_fedTokens` で**最長共通接頭辞を `RewindTo`+再利用**
- `system + tools + 履歴` の prefill は**初回/暖機の1回に償却**
- → turn2 以降は再利用、`/clear` 後の prefill も実測 147→186ms と小さい

> 安定プレフィックスは常駐KVに載りっぱなしで使い回すのが設計。
> `/clear` でそれを壊してはいけない。

---

## 3バックエンド比較（Qwen3-4B・CPU）

llama.cpp（LLamaSharp / GGUF）を2つ目のバックエンドに追加し実測:

| | decode tok/s | prefill(cold) | モデルロード |
|---|--:|--:|--:|
| ONNX int4 | ~7.8–8.0 | 24.6 | ~17s |
| **GGUF Q4_K_M** | ~7.8–8.0 | **29.7** | **~6–7s** |
| GGUF Q5_K_M | 7.1 (~9%遅) | 14.4 | — |

- 正確さ（ハーネス25・R=1）は 21/20/20 ＝ ばらつき範囲内で**優劣つかず**
- **結論：CPU では GGUF Q4_K_M が実質ベスト**（decode 互角・ロード/prefill 上・2.5GB）
- **Q5_K_M は CPU では割に合わない**（prefill 約2倍・精度の上積み無し）

> 「量子化を上げる」より「評価のR増」「モデルサイズ」で精度を取りに行く

# 3. 小型モデルの失敗と対策

## 能力はコードに、プロンプトは最小限

---

## 主要な失敗モード

1. **虚偽成功（最重要）**
   - ファイル変更依頼で read だけして「書き換えました」と嘘の完了報告
2. **決定的崩壊 / repetition collapse**
   - 長文生成が短周期ループに落ち、JSON が壊れる
3. **幻覚（特に 1.7B）**
   - `sed` / `grep` / `Edit-File` / `rename_file` など Linux ツールや架空 cmdlet
4. **オフバイワン**（「N行目」を0-baseで解釈）

> 失敗は **精度律速**（JSON崩れ・暴走）と **容量律速**（数値捏造・PowerShell想起）に二分

---

## repetition collapse — 推測せず実測で切り分ける

ORT-GenAI のサンプリングオプションを A/B 実測:

| オプション | 効果 |
|---|---|
| `no_repeat_ngram_size` | **完全に無視される（no-op）** — 使うな |
| `repetition_penalty` | **効く**（1.05→1.15 でループが崩れる） |
| `temperature` | ツール選択安定のため低温維持（反復は増えるが優先） |

- **確実な停止はデコードループの反復ガード**（末尾が短周期で10回続いたら break）
- penalty で発生確率を下げ、ガードを最終保険にする**二段構え**

> 反復不具合は penalty を上げる前に、**オプションが honor されるか実測**せよ

---

## 「能力はコードに、プロンプトは最小限」

決定論的な**コード機構**は monotonic に積み上がる:

- **パーサ補修** — `"key="` → `"key":"`、`"key","` → `"key":"`、無効エスケープの二重化
- **追記をコードで成立**（空 old_string を末尾追記に）→ append 0/3 → 3/3
- **冗長な破壊的上書きガード**（edit_file 直後の write_file 全文上書きをブロック）
- **暴走ループの早期打ち切り**（同一失敗3回で中断）
- **near-match ヒント**（大小文字/改行/エスケープ違いの候補を提示）

一方、**プロンプト微調整は Pareto のシーソー**:

- phi4-mini / Qwen3-4B では **1トークンの変更が別タスクを割る**＝ほぼゼロ和
- ただし **1.7B は failure が categorical** → 欠けた few-shot 例の ROI が高い

---

## 評価ハーネス — 自己申告でなく実ファイルで判定

- `AgentCapabilityHarness`（`RUN_AGENT_HARNESS=1` でゲート）
- GUI 無しで **実エンジン + 実ツール**を駆動、ワークスペースは一時フォルダ
- **モデルの「やりました」ではなく実ファイルの中身で合否判定**（オラクル厳格化が必須）
- ⚠ **「greedy = 再現的」は誤り** → run 間で 7/8/10 とブレる
  - 真因①：パース失敗リトライの温度、②：常駐エンジンのターン跨ぎ状態
  - 真因③（本命）：**genai_config の既定 `top_k=1`** で温度を上げても greedy のまま
  - → **retry 時に `top_k=40` を明示**して候補プールを開く
- **単発のPASS数で前後比較してはいけない → R=3 必須**

---

## 過適合の罠 — 評価とプロンプトの共進化

- 固定15タスクに対し「落ちたタスクを直す1行を足す」を繰り返した
- 結果：**プロンプトと評価セットが共進化**
  - few-shot 例にハーネスのシードファイル名（`README.md`）が混入
  - 個別タスクの対症文が積層
- この状態の「プロンプト変更はゼロ和」は**汎化の証拠ではない**（飽和した過適合評価の artifact）

**対策**

1. プロンプトと語彙の被らない**10タスク追加**（25タスク化・held-out）
2. プロンプトを**一般原則へ汎用化**（ハーネス固有名を全廃）
3. シード名の再混入を**テストで機械的に拒否**

> 実証：汎用プロンプト + 決定論機構3巡で、過適合版と**同点（hard-fail はより少ない）**

---

## エラー可読性が自己修正力を支配する

- **ツールのエラーが読めるか**は、小型モデルの**自己修正力を横断的に左右**する
- 失敗理由がモデルに届けば直せるが、届かなければ「該当なし」と誤読して
  **反復上限まで同じ誤りを再生産**してしまう
- 自己修正が回るには2段が必要:
  1. **エラーが読める**こと（失敗理由が正しくモデルへ渡る）
  2. **次の一手の機械的提示**（決定論ヒント）

> 💡 「読める」だけでも「ヒント」だけでも不十分。両方そろって自己修正が回り、
> 全25タスク R=3 で **25/25** に到達

---

# 4. ツール設計

## なぜツールは4つだけか

- 旧根拠「ツールが多いと毎ターンの prefill が重い」は **Ollama 時代の測定**
- **ONNX では `system+tools` プレフィックスが償却される** → 数個の追加は負担にならない
- 残るコストは **ツール選択の信頼性**（選択肢が増える＝小型モデルが外す＝反復増）

→ ツールは **少数・排他的・動詞名**に保つ

| ツール | 役割 |
|---|---|
| `run_powershell` | 読み・検索・一覧・ビルド・テストすべて |
| `write_file` / `edit_file` | 構造化された引数で**二重エスケープを回避** |
| `web_search` | 見えるブラウザペインで Bing を構造抽出 |

---

## 2つの実行モデル — シングルループ と ステップ実行

同じツール集合・同じ `AgentOrchestrator` を、**2通りの回し方**で使い分ける:

**① シングルループ（チャット / ReAct 型）— `RunTurnAsync`**
- 1ターン＝**1本の連続ループ**。モデルが動的にツールを選び、結果を食べて反復（最大25回）
- ツール呼び出しが無くなったら終了＝**モデル自身が完了を判断**
- 全反復で `system + ツール集合` を**バイト不変**に保ち KVキャッシュを再利用
- 向き：**探索的・一回限り・手順が事前に決まらない**タスク

**② ステップ実行（ワークフロー / plan-then-execute）— `RunStepAsync`**
- タスクを**事前定義のステップ列**に分解し、各ステップを順に実行
- 各ステップは内部で**同じ Orchestrator を1ターン**回す（`turnPreamble` で文脈注入）
- ステップ毎に**出力要約・承認**、最後に最終ステップの出力だけ全文表示
- 向き：**再現したい・多段・チェックポイント/承認を挟みたい**タスク

---

## どちらを選ぶか — 小型モデルでの設計指針

| | シングルループ | ステップ実行 |
|---|---|---|
| 制御の主体 | **モデル**（動的） | **設計者**（事前分解） |
| 判断空間 | 広い（毎反復で全ツール） | 狭い（1ステップ＝1目的） |
| 一発成功率 | 低くなりがち | 上げやすい |
| 計画力の要求 | ループ中に必要 | **分解時に前倒し** |

- 小型モデルは**選択肢が広いほど外す → 反復増 → prefill 再払い**（総時間が伸びる）
- **ステップ分解は1ステップの判断空間を狭め**、一発成功率を上げて無駄な prefill を減らす
- ただし**分解自体に計画力**が要る → 確立した手順はステップ、開いた探索はループ
- どちらも UI は**ステップ単位に畳まずフラットな進捗タイムライン**で統一（`ActivityLog` 共有）

> 「賢いモデルに全部任せる単一ループ」と「人が手順を切るステップ実行」は
> 二者択一でなく、**タスクの確立度で使い分ける両建て**

---

## まとめ — 横断する教訓

- 🎯 **遅延は prefill 律速** — まず内訳（load/prefill/decode）を切り分ける
- 🎯 **総時間は「一発成功率 × prefill」** — decode tok/s だけで選ばない
- 🎯 **弱いGPUは完全に切る**（中途半端なオフロードは逆効果）
- 🎯 **CPU int4 のベストは GGUF Q4_K_M**（量子化を上げても割に合わない）
- 🎯 **能力はコードに、プロンプトは最小限**（決定論機構は積み上がる）
- 🎯 **評価は実ファイルで・R=3 で・held-out で**（greedy ≠ 再現的）
- 🎯 **エラー可読性が自己修正力を支配する**（読める＋次の一手の提示）

> 小型 non-thinking モデルでも、**仕組み（コード・評価・基盤）で堅く積めば**
> CPU 単機でエージェントは実用ラインに乗る
