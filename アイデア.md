# Loomo の方向性アイデア

IDE 機能を単純に足すのではなく、既存資産（Editor / Terminal / Browser / Git / Diff / Search / LSP / Workflow / Trace / AI tools）を「AI と人間が一緒に作業する作業台」として束ねる方向のメモ。案は **方向性**（製品全体の寄せ先）／**独立パーツ**（単体で役立つ新ペイン・専用ビュー）／**横断機能**（既存パーツを束ね、移動と確認を減らす）の3種。`AI_Roadmap.md`・設計書ロードマップに既にある案は正本をそちらとし、ここは組み合わせ検討用の索引とする。

## パーツ間の共通接続

各パーツは可能な範囲で5つの共通操作を持つ: **ここへ送る**（Editor 選択・Terminal 出力・Browser 内容・ファイルを渡す）／**元を開く**（結果から元ファイル・行・URL・Terminal タブへ戻る）／**比較する**（Diff・Data Diff・Visual Diff の適切な比較器へ）／**残す**（Pegboard へ）／**実行する**（見える Terminal で）。共通の巨大なメッセージ基盤を先に作らず、最初のパーツに必要な最小の受け渡し型から始める。

## 優先候補

**パーツ優先**: ① Log Viewer＋Stack Trace Navigator（Editor–Terminal 間を埋め用途明確）② Data Lab＋Data Diff（CSV/JSON 表示を閲覧→操作へ）③ Visual Diff＋Web Capture（内蔵 Browser の強み）④ Process Tree＋Port Inspector（Terminal で動かしたものを別視点で管理）⑤ Encoding Inspector（Windows 開発で頻出）⑥ Transform Tray（小さく始め全ペインから利用可）。この方向は Loomo を「開発中の異なる素材を専用道具で扱い、相互に受け渡せるワークベンチ」として育て、IDE 機能追加とも Agent OS 構想とも両立する。

**方向性優先**: Agent OS ＋ Local Automation Studio ＋ AI Change Management の組み合わせ。Loomo を「コードを書くエディタ」ではなく「AI のローカル作業を人間が監督・編集・承認するデスクトップアプリ」として育てる。

## 上位概念：道具の集合から意味的な作業空間へ

各案に共通するのは「異なる素材を受け取り、加工し、別の場所へ渡し、出自と結果を残す」こと。そこで Loomo の主役をペインやファイルから、仕事とその意味的構造へ移す。構想は **Work Fabric**（素材と加工の流れを一つの仕事として記録）→ **Living Workpiece**（素材を運ぶのでなく、仕事の対象を中心に各ペインが反応）→ **Work Volume**（ペイン配置に意味的な奥行きを追加）→ **Semantic Depth**（奥行きの意味を「理解の解像度」に限定）と段階的に発展した。

### Work Fabric

仕事から生まれる素材と関係を一枚の織物として扱う（例: 目的 → Web 調査 → ログ採取 → StackTrace 解析 → コード特定 → 変更 → テスト → Visual Diff 確認 → コミット）。基本語彙:

| 概念 | 意味 |
|---|---|
| Thread | 「この不具合を直す」など目的を持つ作業の糸 |
| Material | 出自と型を持ち、ペイン間で扱える入力・生成物 |
| Lens | 同じ Material を見るための異なる表示・操作方法 |
| Operation | Material を別の Material へ変換する人間・CLI・AI の操作 |
| Evidence | 変更や判断の正しさを支える検証結果 |
| Weave | Thread・Material・Operation・Evidence から成る仕事全体 |
| Pattern | 成功した Weave から抽出する再利用可能な仕事の型 |

Material の型は `Text` `File` `CodeRange` `Url` `Command` `Log` `StackTrace` `Dataset` `Image` `Diff` `TestResult` `Process` `Diagnosis` など。クリップボードと違い、元の場所・作成時刻・生成方法・所属 Thread を保持する。同じ Material を複数の Lens で見られ（JSON なら Text 編集 / Tree 閲覧 / Diff 比較 / Query 抽出）、Operation は型付き変換（`Log→StackTrace`、`StackTrace→CodeRange`、`Screenshot×Screenshot→VisualDiff`、`GitChanges＋TestResult→VerificationResult`）として表せる。この語彙は設計を考える道具であり、共通基盤として先に実装しない。最初に実装へ拾うのは、Breadcrumb の軌跡を出自付きで残すジャンプ履歴（Thread Rail の種）だけ。

初期 UI 案は5つ: **Material Handle**（選択・出力・要素を出自付き Material としてつかむ）、**Thread Rail**（現在の作業を画面下部の細い流れとして表示。展開で素材・操作・分岐・検証結果が見える）、**Shuttle**（現在の Material に適用可能な Operation だけを表示。自然言語指示も同じ場所で扱う）、**Lens Stack**（タブを増やさず同一 Material の見方 Text/Tree/Query/Diff を重ね替え）、**Weave View**（分岐・出自・Evidence・却下案の俯瞰）。

### Living Workpiece

Material を別の道具へ運ぶ操作もまだアプリ中心の発想。さらに進め、中心を「ユーザーが今取り組んでいる対象」に置く（例:「ログイン失敗」＝ 症状: 500応答 / 観測: server.log の例外 / 場所: AuthService.cs:84 / 仮説: null user / 変更: null check 追加 / 根拠: test 成功・画面確認）。Editor/Terminal/Browser/Tests は別素材を開く独立した箱ではなく、同じ Workpiece を異なる角度から見る Lens になる。

Terminal で例外へ注目すると Loomo 全体の **Semantic Focus** がそこへ移る——「どこへ送るか」ではなく「何に注目しているか」に全体が反応する。ただし各ペインの表示を勝手に書き換えず、Editor の該当行・FolderTree の関連ファイル・関連テスト・Browser の対応リクエスト・Git の関連変更は袖の候補として現れ、ユーザーが選んだときだけ中央に実体化する。AI もチャットボックスに閉じず Workpiece を観察して未確認の関係や候補を UI 上へ提示するが、描画で区別する: 事実=実線、AI の推測=半透明、未確認の関係=点線、反証済み=減光、検証済み=固定表示。

### Work Volume と Semantic Depth：奥行きは解像度

2次元のペイン配置に意味的な奥行きを加える。実際の 3D モデル回転ではなく、カードの重なり・縮小・減光・半透明・軽いパララックス・Semantic Zoom による 2.5D。初期案は過去・仮説・確度・分岐をすべて Z 軸に割り当てたが、一本の軸に複数の意味があると空間として理解できない。そこで Z 軸は「対象をどこまで深く理解しているか」という解像度だけを表す:

```text
浅い ← Workspace → Workpiece → Artifact → Symbol → Execution → Value → 深い
```

ズームで文字が拡大されるのではなく、見える概念が変わる（地図の 国→都市→道路→建物→店内 と同じ縮尺の変化）。ただし航行できる深度は **File / Symbol / Execution の3段から始める**。Workspace/Workpiece は Breadcrumb 左端の文脈表示に留め、Value は Execution Lens 内の変数ツリーとして扱う。データの無い深度は「潜ったら空だった」を生み、一度でも起きると奥行きメタファーへの信頼を壊すため、実データと Lens が揃った深度だけを開通させる。

例: `Login()` へ潜ると本文だけでなく呼び出し元・関連テスト・直近の変更・実行時例外が見える。特定の実行へ潜ると引数・戻り値・所要時間・HTTP request。値へ潜ると `user = null ← repository.Find("koya") ← SQL 0 rows ← connection: Development ← appsettings.Development.json` と出自を遡れる（遡行には計装が必要なため、独立した深度への昇格は目処が立ってから）。体験の中心:

> エラーからコードへ移動するのではなく、エラーへ潜ったら、その内側にコードがあった。

同様に、関数へ潜ったら関連テストが同じ空間にある。**Focus Well** はこのメンタルモデル（中央へ近づくほど詳細、外へ戻るほど全体との関係が見える同心円）だが、同心円は頭の中の像であって画面レイアウトではない。「潜った」感覚は、選択要素が拡大して中央へ移る遷移アニメーションで作る。

操作は意味を分離する: ダブルクリック/ホイール上=一段潜る、ホイール下/`Esc`=一段戻る、左右=同深度の関連対象へ、Lens 切替=見る角度（Code/Runtime/Tests/Logs/History）、Timeline=同一対象の時間変化、俯瞰=Workpiece 全体。時間は横方向の連続、仮説は同深度から分かれる枝、事実/推測/検証状態は座標ではなく線種・透明度・色で表し、いずれも Z 軸とは分離する。

### UI として成立させる制約

Work Volume 全体を常時描画すると巨大ノードグラフか操作しにくい 3D デモになる。通常画面は現在の一断面と一段外・一段内だけを見せ、UI 要素は4つに絞る: ① **Depth Breadcrumb**（現在地と帰り道、例 `Project > Issue > File > Login() > Run #42`）② **中央の既存ペイン**（現在の断面）③ **Lens Strip**（同じ対象を見る角度の切替）④ **Depth Navigation**（親へ戻る/対象へ潜る）。

守るべき原則:

- **中央を勝手に動かさない**: AI や新しいログは袖に候補として出し、ユーザー選択時のみ中心を移す。
- **一操作で一軸だけ動かす**: 深度移動・同深度移動・Lens 切替・時間移動を同時に行わない。
- **帰り道を常に示す**: `Esc` で一段、Breadcrumb で任意階層、履歴で直前の視点へ戻れる。
- **全情報を同時に出さない**: 全ノード・全時系列・全仮説・全 Lens・AI の全推論は俯瞰か一時展開に限定。
- **移動の連続性を保つ**: 拡大遷移アニメーションで「別画面を開いた」ではなく「対象へ潜った」と感じさせる。
- **空の深度・Lens を見せない**: データの無い Lens は消さず、「このテストを実行すると見られる」「デバッグをアタッチする」など入手手段を示す。「潜れば必ず何かある」という信頼がメタファーの前提。
- **AI を移動の経路に入れない**: 関連候補の列挙は LSP・Git・パーサなど決定論的情報源で即時に行う。ローカル LLM は非同期の後追い（候補への注釈・仮説の提示）に限定し、潜る/戻るの応答時間に関与させない。

既存 UI は捨てず役割を再定義する: **舞台**=Work Volume の現在断面、**袖**=現在面に関連する一段外・一段内の候補、**奥**=より浅い文脈または現在に至る出自、**俯瞰**=Workpiece/Weave 全体。ステージモードは「立体的な仕事のどの断面を見るか」を表す UI になる。

### 段階的な検証

大規模グラフや 3D UI を先に実装せず、一つの移動が自然かを検証する。

- **実験A（Material の受け渡し）**: Terminal 出力を Material Handle でつかむ → Shuttle にその型で可能な Operation を表示 → 「スタック位置を開く」で Editor へ → Thread Rail に `Log → StackTrace → CodeRange` が残る。
- **実験B（Semantic Depth）**: Terminal のスタックフレームをクリック → 対応メソッドが中央へ連続拡大 → Breadcrumb に `Log > Stack Frame > Login()` → Lens を Runtime に切替え引数・例外を表示 → `Esc` で同じ経路を戻る。Lens Strip は既存資産に固定: Code（Editor）、History（Git Blame→Diff 該当行表示、実装済）、Runtime（netcoredbg/DAP 導入済。セッションが無ければ入手手段を表示）。新規に作るのはスタックトレースパーサ・遷移アニメーション・Breadcrumb/Lens Strip の枠だけ。
- **実験C（Living Workpiece）**: 例外から Workpiece を自動生成 → Editor/Git/Tests の関連対象が袖に候補として現れる → 修正を Change、テスト結果を Evidence として接続 → Evidence が揃うと解決済みに。

実験B の一往復が自然なら、同じ操作を Task/File/Test/Browser/Runtime Value へ広げる。ここが重ければ、全体グラフを実装しても日常利用されない。

### 暫定的な製品定義

概念名は **Semantic Depth** とする（Focus Well は同心円 UI を採らないため部品名として残さない。Loom Space / Work Volume / Semantic Workspace は対外名称の候補として保留）。

> Loomo は、仕事・コード・実行・値の間を意味的にズームできる、反応型の開発空間である。

Work Fabric まで含めた広い定義:

> 人間・AI・ツールが生み出す素材を、出自を保ったまま加工し、仕事の織物として理解・探索できる環境。

まだ UI イメージの検証段階にあり、「奥行きが自然に感じられるか」「既存ペインより速く目的へ到達できるか」「自動反応が注意を奪わないか」をプロトタイプで確認する必要がある。
