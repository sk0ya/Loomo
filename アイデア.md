# Loomo の方向性アイデア

Loomo を IDE として強化する以外の方向性メモ。

## 前提

Loomo にはすでに Editor / Terminal / Browser / Git / Diff / Search / LSP / Workflow / Trace / AI tools がある。

そのため、単に IDE 機能を足していくよりも、これらを「AI と人間が一緒に作業するための作業台」として束ねる方向が独自性を出しやすい。

## 1. Agent OS

複数の AI エージェントを動かす司令室としての方向。

- エージェントごとの役割、権限、作業ディレクトリ
- 並列タスク実行
- タスクキュー
- 失敗時の再試行ポリシー
- エージェント同士の引き継ぎログ
- 人間の承認待ち一覧

Editor や Terminal は主役ではなく、エージェント作業を観察・介入するための部品として扱う。

## 2. Local Automation Studio

ローカル PC 上の自動化ワークフロー環境としての方向。

- ファイル監視トリガー
- 時刻指定実行
- コマンド、AI、ファイル変換、Web 検索のステップ化
- 実行履歴と再実行
- 失敗時通知
- 定型作業テンプレート

Zapier / n8n 的なものを、ローカルファイル、Terminal、AI に強くした形。

## 3. Research / Knowledge Workspace

調査・読解・要約・比較のための環境としての方向。

- Web / PDF / Markdown / repo docs の収集
- 出典付きノート
- 要約比較
- 調査セッション管理
- 質問ごとの根拠リンク
- 知識カード化

コードを書くより、調べて理解して判断するための作業空間。

## 4. AI Change Management

AI が加えた変更を管理するシステムとしての方向。

- AI 作業単位ごとの変更セット
- なぜ変えたか、何を読んだか、どのテストをしたかの記録
- 変更の承認、破棄、一部採用
- 複数案の比較
- ロールバック
- 変更履歴からの学習

Git より細かい「AI 作業ログ + diff 管理」に寄せる。

## 5. Personal Ops Console

開発に限らず、PC 内作業をまとめる個人用コンソールとしての方向。

- よく使うフォルダ、プロジェクト、スクリプト
- ローカル LLM 起動管理
- バックアップ、同期、変換処理
- ログ監視
- 日次ルーチン
- メモ、チェックリスト

自分専用の作業管制塔。

## 6. Local AI Lab

ローカル LLM 実験環境としての方向。

- モデル比較
- プロンプト評価
- 推論速度計測
- コンテキスト長や量子化違いの比較
- 回答品質ベンチ
- ツール呼び出し成功率の測定

既存の local inference や harness report と相性がよい。

## アイデアの分類

以降の案は、性質を次の3種類に分ける。

- **方向性**: 製品全体をどこへ寄せるか。上記1〜6が該当する。
- **独立パーツ**: 単体でも役立つ、新しいペインまたは専用ビュー。
- **横断機能**: 既存パーツを束ね、作業の移動や確認を減らす機能。

`AI_Roadmap.md` や設計書のロードマップに既にある案は実装計画の正本をそちらとし、このファイルでは組み合わせを考えるための索引として扱う。

## パーツ間の共通接続

新しいパーツを孤立させないため、各パーツは可能な範囲で次の共通操作を持つ。

- **ここへ送る**: Editor選択範囲、Terminal出力、Browser内容、ファイルを現在のパーツへ渡す。
- **元を開く**: 結果から元ファイル・行・URL・Terminalタブへ戻る。
- **比較する**: 現在の素材をDiff、Data Diff、Visual Diffの適切な比較器へ渡す。
- **残す**: 結果や参照をPegboardへ送る。
- **実行する**: 必要なコマンドを見えるTerminalで実行する。

共通の巨大なメッセージ基盤を先に作るのではなく、最初のパーツで必要な最小の受け渡し型から始める。

## 優先候補

### 新しいパーツを増やす場合

1. **Log Viewer + Stack Trace Navigator** — EditorとTerminalの間を埋め、用途が明確。
2. **Data Lab + Data Diff** — 既存のCSV/JSON表示を閲覧から操作へ発展できる。
3. **Visual Diff + Web Capture** — 内蔵Browserを持つLoomoの強みが出る。
4. **Process Tree + Port Inspector** — Terminalで動かしたものを別の視点から管理できる。
5. **Encoding Inspector** — Windows開発で頻出する文字コード問題に強くなる。
6. **Transform Tray** — 小さく始められ、全ペインから利用できる。

### 製品全体の方向性として組み合わせる場合

- Agent OS
- Local Automation Studio
- AI Change Management

この組み合わせでは、Loomoを「コードを書くエディタ」ではなく、AIがローカル環境で作業するのを人間が監督・編集・承認するデスクトップアプリとして育てる。

一方、独立パーツを優先する場合は、Loomoを「開発中に発生する異なる素材を、適切な専用道具で扱い、相互に受け渡せるワークベンチ」として育てる。この方向は通常のIDE機能追加ともAgent OS構想とも両立できる。

## 上位概念：道具の集合から意味的な作業空間へ

個別パーツと横断機能をまとめて考えると、共通しているのは「異なる素材を受け取り、加工し、別の場所へ渡し、その出自と結果を残す」ことである。ここから、Loomoの主役をペインやファイルではなく、仕事とその意味的な構造へ移す。

この構想は段階的に次のように発展した。

1. **Work Fabric** — 素材と加工の流れを一つの仕事として記録する。
2. **Living Workpiece** — 素材を運ぶのではなく、仕事の対象を中心に各ペインが反応する。
3. **Work Volume** — 2次元のペイン配置へ、意味的な奥行きを加える。
4. **Semantic Depth** — 奥行きの意味を「対象を理解する解像度」に限定する。

### Work Fabric

仕事から生まれる素材と関係を、一枚の織物として扱う。

```text
ユーザーの目的
  → Webで調査
  → ログを採取
  → Stack Traceを解析
  → 関連コードを特定
  → ファイルを変更
  → テストを実行
  → Visual Diffで確認
  → コミット
```

基本語彙は次のとおり。

| 概念 | 意味 |
|---|---|
| Thread | 「この不具合を直す」など、目的を持った作業の糸 |
| Material | 出自と型を持ち、ペイン間で扱える入力・生成物 |
| Lens | 同じMaterialを見るための異なる表示・操作方法 |
| Operation | Materialを別のMaterialへ変換する人間・CLI・AIの操作 |
| Evidence | 変更や判断が正しいことを支える検証結果 |
| Weave | Thread、Material、Operation、Evidenceから成る仕事全体 |
| Pattern | 成功したWeaveから抽出する再利用可能な仕事の型 |

Materialの型には `Text`、`File`、`CodeRange`、`Url`、`Command`、`Log`、`StackTrace`、`Dataset`、`Image`、`Diff`、`TestResult`、`Process`、`Diagnosis` などがある。単なるクリップボードと違い、元の場所、作成時刻、生成方法、所属Threadを保持する。

同じMaterialを複数のLensで見られる。

```text
JSON Material
  ├─ Text Lens   → テキストとして編集
  ├─ Tree Lens   → 構造として閲覧
  ├─ Diff Lens   → 別のJSONと比較
  └─ Query Lens  → JSONPathで抽出
```

Operationは型を持った変換として表せる。

```text
Log → StackTrace
StackTrace → CodeRange
BrowserPage → Screenshot
Screenshot × Screenshot → VisualDiff
GitChanges + TestResult → VerificationResult
```

この語彙は設計を考えるための道具であり、この体系を共通基盤として先に実装しない（「共通の巨大なメッセージ基盤を先に作らない」方針と同じ）。最初に実装へ拾うのは、Breadcrumbの軌跡を出自付きで残すジャンプ履歴（Thread Railの種）だけとする。

### Material Handle / Thread Rail / Shuttle

Work Fabricを直接操作する初期UI案。

- **Material Handle**: Editorの選択、Terminal出力、Browser要素などを、出自付きMaterialとしてつかむハンドル。
- **Thread Rail**: 現在の作業を画面下部に細い流れとして表示する。展開すると素材、操作、分岐、検証結果が見える。
- **Shuttle**: 現在のMaterialへ適用できるOperationだけを表示する操作器。自然言語による指示も同じ場所で扱う。
- **Lens Stack**: 新しいタブを増やさず、同じMaterialに対する `Text / Tree / Query / Diff` などの見方を重ね替える。
- **Weave View**: 作業全体の分岐、出自、Evidence、却下案を俯瞰する。

```text
┌──────────────────────────────────────────────┐
│ Project / branch / current Thread            │
├─────────┬───────────────────────┬────────────┤
│ Sidebar │        Stage          │    袖      │
│         │ Material Handle       │ Lens候補   │
├─────────┴───────────────────────┴────────────┤
│ Thread Rail: [Log] → [Code] → [Patch] → [Test]│
├──────────────────────────────────────────────┤
│ Shuttle: 選択素材への操作／自然言語          │
└──────────────────────────────────────────────┘
```

### Living Workpiece

Materialを別の道具へ運ぶ操作も、まだアプリケーション中心の発想である。さらに進め、Loomoの中心を「ユーザーが今取り組んでいる対象」にする。

```text
ログイン失敗
├─ 症状: 500応答
├─ 観測: server.logの例外
├─ 場所: AuthService.cs:84
├─ 仮説: null user
├─ 変更: null check追加
├─ 実行: localhost:5173
└─ 根拠: test成功、画面確認
```

Editor、Terminal、Browser、Testsは別の素材を開く独立した箱ではなく、同じWorkpieceを異なる角度から見るLensになる。

Terminalで例外へ注目すると、Loomo全体の **Semantic Focus** がその例外へ移る。「どこへ送るか」ではなく、「何に注目しているか」に全体が反応する。ただし各ペインの表示を勝手に書き換えるのではなく、Editorの該当行、FolderTreeの関連ファイル、Testsの関連テスト、Browserの対応リクエスト、Gitの関連変更は袖の候補として現れ、ユーザーが選んだときだけ中央に実体化する（後述「中央を勝手に動かさない」の原則をペイン反応にも適用する）。

AIもチャットボックスの中だけに置かない。Workpieceを観察し、未確認の関係や候補をUI上へ提示する。

- 事実: 実線
- AIの推測: 半透明
- 未確認の関係: 点線
- 反証済み: 減光
- 検証済み: 固定表示

AIが勝手に現在面を変更せず、候補を周辺へ置き、ユーザーが触れたときに実体化させる。

### Work Volume

既存Loomoは、ペインをX/Y方向に配置する2次元の作業空間である。そこへ意味的な奥行きを加える。ただし、実際の3Dモデルを回転するUIにはしない。カードの重なり、縮小、減光、半透明、軽いパララックス、Semantic Zoomで奥行きを感じさせる2.5D UIとする。

初期案では、過去・仮説・確度・分岐をすべてZ軸に割り当てた。しかし一本の軸に複数の意味があると、ユーザーが空間として理解できない。このため奥行きの意味を一つに限定する。

### Semantic Depth：奥行きは解像度

Z軸は「対象をどこまで深く理解しているか」という意味的な解像度だけを表す。

```text
浅い
  Workspace
    ↓
  Workpiece
    ↓
  Artifact
    ↓
  Symbol
    ↓
  Execution
    ↓
  Value
深い
```

ズームして文字が拡大されるのではなく、見える概念が変わる。

```text
Project → Task → File → Symbol → Execution → Value
```

地図の `国 → 都市 → 道路 → 建物 → 店内` と同じく、同じ対象を異なる縮尺で見る。

ただし航行できる深度は **File / Symbol / Execution の3段から始める**。WorkspaceとWorkpieceはBreadcrumb左端の文脈表示に留め、ValueはExecution Lens内の表示（変数ツリー）として扱う。データの無い深度は「潜ったら空だった」を生み、一度でも起きると奥行きのメタファーへの信頼を壊すため、実データとLensが揃った深度だけを開通させる。

例として `Login()` へ潜ると、単なるソース本文ではなく、呼び出し元、関連テスト、直近の変更、実行時の例外が見える。さらに特定の実行へ潜ると引数、戻り値、所要時間、HTTP requestが見える。値へ潜ると、その値がどのSQL結果や設定から生まれたかを遡れる（この遡行は計装が必要なため、独立した深度への昇格は目処が立ってからとする）。

```text
user = null
  ← repository.Find("koya")
      ← SQL result: 0 rows
          ← connection: Development
              ← appsettings.Development.json
```

この体験の中心は次の表現に集約される。

> エラーからコードへ移動するのではなく、エラーへ潜ったら、その内側にコードがあった。

同様に、コードからテストを探すのではなく、関数へ潜ったら関連テストが同じ空間にある。

### Focus Well

Semantic Depthを表すメンタルモデル。画面中央へ近づくほど詳細になり、外へ戻るほど全体との関係が見える。ただし下図の同心円は頭の中の像であって画面レイアウトではない。実UIは後述の4要素（Breadcrumb・中央ペイン・Lens Strip・Depth Navigation）で構成し、「潜った」という感覚は同心円の描画ではなく、選択要素が拡大して中央へ移る遷移アニメーションで作る。

```text
              周辺コンテキスト
    ┌─────────────────────────┐
    │      関連ファイル       │
    │   ┌─────────────────┐   │
    │   │  関連シンボル   │   │
    │   │  ┌───────────┐  │   │
    │   │  │ Login()   │  │   │
    │   │  │    ↓      │  │   │
    │   │  │ user=null │  │   │
    │   │  └───────────┘  │   │
    │   └─────────────────┘   │
    └─────────────────────────┘
                  深い
```

操作の意味を分離する。

- ダブルクリック／ホイール上: 対象へ一段潜る。
- ホイール下／`Esc`: 一段外へ戻る。
- 左右移動: 同じ深度にある関連対象へ移る。
- Lens切替: Code、Runtime、Tests、Logs、Historyなど、見る角度を変える。
- Timeline: 同じ対象の時間的な変化を見る。
- 俯瞰: 現在地点を含むWorkpiece全体を見る。

時間、仮説、事実性はZ軸と分離する。

- **時間**: 同じ対象の横方向の連続。
- **仮説**: 同じ深度から分かれる枝。
- **事実／推測／検証状態**: 座標ではなく線種、透明度、色などの描画方法。

### UIとして成立させる制約

Work Volume全体を常時描画すると、巨大なノードグラフか操作しにくい3Dデモになる。通常画面では現在の一断面と、その一段外・一段内だけを見せる。

```text
┌ Project > Issue > File > Login() > Run #42 ┐
├─────────────────────────────────────────────┤
│                                             │
│        現在のLens: Editor / Runtime         │
│                                             │
├─────────────────────────────────────────────┤
│ Code  Runtime  Tests  Logs  History         │
├─────────────────────────────────────────────┤
│ ← 親へ     関連: 6      仮説: 2      深く → │
└─────────────────────────────────────────────┘
```

必要なUI要素は4つに絞る。

1. **Depth Breadcrumb** — 現在地点と帰り道を示す。
2. **中央の既存ペイン** — 現在の断面を表示する。
3. **Lens Strip** — 同じ対象を見る角度を切り替える。
4. **Depth Navigation** — 親へ戻る／対象へ潜る。

守るべき原則:

- **中央を勝手に動かさない**: AIや新しいログは候補として袖に出し、ユーザー選択時だけ中心を移す。
- **一操作で一軸だけ動かす**: 深度移動、同一深度移動、Lens切替、時間移動を同時に行わない。
- **帰り道を常に示す**: `Esc`で一段戻る、Breadcrumbで任意階層へ戻る、履歴で直前の視点へ戻る。
- **全情報を同時に出さない**: 全ノード、全時系列、全仮説、全Lens、AIの全推論は俯瞰または一時展開に限定する。
- **移動の連続性を保つ**: 選択要素が拡大して中央へ移るアニメーションにより、「別画面を開いた」ではなく「対象へ潜った」と感じさせる。
- **空の深度・Lensを見せない**: データの無いLensは消すのではなく、「このテストを実行すると見られる」「デバッグをアタッチする」など入手手段を示す。潜れば必ず何かある、という信頼がメタファーの前提になる。
- **AIを移動の経路に入れない**: 関連候補の列挙はLSP・Git・パーサなど決定論的な情報源で即時に行う。ローカルLLMは非同期の後追い（候補への注釈、仮説の提示）に限定し、潜る／戻るの応答時間に関与させない。

### 既存のステージ／袖／俯瞰との統合

既存UIを捨てず、役割を再定義する。

```text
舞台 = Work Volumeの現在断面
袖   = 現在面に関連する一段外・一段内の候補
奥   = より浅い文脈または現在に至る出自
俯瞰 = Workpiece / Weave全体
```

ステージモードは単なるレイアウトではなく、「立体的な仕事のどの断面を見るか」を表すUIになる。

### 段階的な検証

大規模なグラフや3D UIを先に実装せず、一つの移動が自然かを検証する。

#### 実験A：Materialの受け渡し

1. Terminal出力をMaterial Handleでつかむ。
2. Shuttleに、その型で可能なOperationを表示する。
3. 「スタック位置を開く」でEditorへ移る。
4. Thread Railに `Log → StackTrace → CodeRange` が残る。

#### 実験B：Semantic Depth

1. Terminalのスタックフレームをクリックする。
2. 対応メソッドが中央へ連続的に拡大される。
3. Breadcrumbに `Log > Stack Frame > Login()` を表示する。
4. LensをRuntimeへ切り替え、引数・例外を表示する。
5. `Esc`で同じ経路をログへ戻る。

Lens Stripは既存資産に固定する: Code（Editor）、History（Git Blame→Diffの該当行表示、実装済み）、Runtime（netcoredbg/DAP、導入済み。セッションが無ければ入手手段を表示）。新規に作るのはスタックトレースパーサ、遷移アニメーション、Breadcrumb / Lens Stripの枠だけとなる。

#### 実験C：Living Workpiece

1. 例外からWorkpieceを自動生成する。
2. Editor、Git、Testsの関連対象が袖に候補として同時に現れる。
3. 修正をChangeとしてWorkpieceへ接続する。
4. テスト結果をEvidenceとして接続する。
5. Evidenceが揃うとWorkpieceを解決済みにする。

実験Bの一往復が自然なら、同じ操作をTask、File、Test、Browser、Runtime Valueへ広げる。ここが重ければ、全体グラフを実装しても日常利用されない。

### 暫定的な製品定義

概念名は **Semantic Depth** とする。Focus Wellは同心円UIを採らないため部品名としては残さない。Loom Space / Work Volume / Semantic Workspaceは、対外的な名称を決める段階の候補として保留する。

現時点で最も具体的な定義は次のとおり。

> Loomoは、仕事・コード・実行・値の間を意味的にズームできる、反応型の開発空間である。

また、Work Fabricまで含めた広い定義は次のとおり。

> 人間・AI・ツールが生み出す素材を、出自を保ったまま加工し、仕事の織物として理解・探索できる環境。

この構想はまだUIイメージの検証段階にある。特に「奥行きが自然に感じられるか」「既存ペインより速く目的へ到達できるか」「自動反応がユーザーの注意を奪わないか」をプロトタイプで確認する必要がある。
