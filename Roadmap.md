# Loomo Roadmap

ローカルLLMエージェントとしての実用性を上げるための追加候補。現状の基盤（WPF UI、Ollama/OpenAI互換クライアント、ツール実行、承認、安全ポリシー、トレース分析）を前提に、効果が大きい順に整理する。

## Priority 1: 開発作業の精度を上げる

### ワークスペース検索ツール

- `search_files`: `rg` 相当の全文検索。
- `find_files`: ファイル名・拡張子・パターン検索。
- `find_references`: シンボル名や文字列の参照箇所一覧。

現状は `list_directory` と `read_file` に寄っているため、AIが対象ファイルを探すコストが高い。コード修正エージェントとしては最優先の追加。

### 局所編集ツール（部分実装済み）

- `apply_patch`: SEARCH/REPLACE ブロックで複数箇所を一度に局所適用。（実装済み）
- `replace_range`: 行範囲指定で置換。（実装済み）
- `replace_text_once`: 期待する旧文字列が一意に一致した場合だけ置換。（実装済み）

現状の `propose_edit` はファイル全文を提示する方式なので、大きなファイルではコンテキスト消費と誤変更リスクが高い。全文置換は残しつつ、通常修正は局所編集に寄せる。

### 選択範囲取得（実装済み）

- `get_selection_text`: エディタの現在選択範囲を返す。（実装済み）
- `replace_selection`: 選択範囲に対する編集案を提示・適用する。（実装済み）

`EditorService.GetSelectedTextAsync()` は `VimEditorControl.SelectedText` に接続済み。`replace_selection` はアクティブファイルの現在内容に対して選択テキストが一意に一致する場合だけ置換し、差分提示後に適用する。「この選択範囲を直して」という自然な操作に対応した。

## Priority 2: UIをAIの道具として広げる

### タブ操作ツール

- `list_tabs`: Editor / Terminal / Browser のタブ一覧。
- `activate_tab`: 指定タブを前面へ。
- `close_tab`: 指定タブを閉じる。
- `open_file_or_url`: ファイルパス、URL、検索語を判定して適切なペインで開く。

UI側にはタブ基盤があるため、AIから操作できるようにすると「必要な資料やファイルを開きながら作業する」体験に近づく。

### ブラウザ操作ツール

- `open_url`: Browser ペインでURLを開く。
- `get_page_text`: 現在ページの本文テキストを取得。
- `search_web`: 明示的に許可された場合だけWeb検索する。

ローカル優先の設計を保ちつつ、ユーザーが許可した調査タスクではブラウザペインをAIが使えるようにする。

### Zenモード / フォーカス操作

- `focus_pane`: 指定ペインへフォーカス移動。
- `maximize_pane`: 単一ペインを一時的に最大化。

長い作業中に、AIがユーザーへ見せたいペインを前面化できるようにする。

## Priority 3: 安全性と運用性

### 権限プロファイル

- Read-only: ファイル読み取りと検索のみ。
- Workspace-write: ワークスペース内の編集だけ許可。
- Command-limited: 許可リスト内のコマンドだけ実行。
- Full-manual: 破壊的操作・ネットワーク・ワークスペース外アクセスは毎回承認。

現状は承認とブロックリストが中心。ローカル環境を直接操作するため、モード別にできることを明確に制限したい。

### コマンド実行のサンドボックス強化

- ワークスペース外への書き込み検知。
- ネットワーク使用の明示承認。
- 長時間実行コマンドのタイムアウトと停止UI。
- 環境変数や秘密情報をログへ残さないフィルタ。

安全ポリシーを正規表現ブロックから、実行前後の環境制御へ広げる。

### トレースのプライバシー管理

- トレース保持日数・最大サイズの設定。
- ツール出力やファイル内容のマスク。
- セッション単位の削除。
- エクスポート前の秘匿情報スキャン。

トレース分析は有用だが、プロンプト、ファイル内容、コマンド出力を含むため管理機能が必要。

## Priority 4: 記憶と改善サイクル

### 長期記憶 / プロジェクト知識ベース

- ユーザー方針、プロジェクト固有ルール、過去の解決策を保存。
- embeddings + SQLite などでローカル検索。
- セッション開始時に関連知識を自動注入。

会話履歴だけでなく、再利用可能な知識として蓄積する。ローカルLLMの強みであるプライベート知識の活用につながる。

### 正確な usage / token 記録

- 各AIクライアントで input/output token を抽出。
- トレースへ `ai.usage` を記録。
- 分析UIでモデル別・セッション別の消費を表示。

分析UIは受け皿があるため、クライアント側の usage 抽出を追加すると改善判断に使える。

### プレイバック

- ツールを再実行しない読み取り専用再生。
- ステップ実行、速度変更、タイムライン表示。
- 失敗ターンからのフォーク実行は将来機能として分離。

トレースを「あとで読むログ」から「動作を再現して理解する画面」に引き上げる。

## Priority 5: 拡張性

### MCP / プラグイン式ツール拡張

- MCPクライアントとして外部ツールを接続。
- 独自 `IAgentTool` をアプリ再ビルドなしで追加できる仕組み。
- ツールごとの権限、説明、承認要否を設定UIで管理。

現状はコード上のDI登録で拡張できる。ユーザーや外部ツール作者が追加できる形にすると、ローカルエージェントとして伸びやすい。

### マルチエージェント

- Planner / Coder / Reviewer / Tester などの役割分担。
- 各エージェントの状態を表示。
- 同一トレースへ `agentId` 付きで記録。

まずは単一エージェントを安定させ、トレースと権限管理を土台にしてから段階導入する。
