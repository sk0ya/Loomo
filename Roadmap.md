# Loomo Roadmap

ローカル LLM エージェントとしての実用性を上げるための追加候補。現状の基盤（WPF UI、in-process ONNX 推論、3 ツール体制 `run_powershell` / `write_file` / `edit_file`、承認・安全ポリシー、JSONL トレース、実機精度ハーネス）を前提に、効果が大きい順に整理する。

> 実装状況の最終確認: 2026-06-15。Priority 1〜5 の項目はいずれも未着手（コンテキストはトリムのみ・推論は CPU のみ・ツールは 3 つのまま・MCP / 長期記憶 / プレイバック / 権限プロファイルなし）。

## これまでの主な到達点（完了）

- **ローカル推論の完全 in-process 化**: Ollama を撤去し ONNX Runtime GenAI（CPU int4）へ移行。モデル常駐でロードは初回のみ、KV プレフィックスキャッシュが効く構成に。
- **マルチモデル対応**: Phi-4-mini / Qwen3-1.7B / Qwen3-4B のダウンロード・切替。モデル別プロファイル（コンテキスト窓・サンプリング・チャット書式）。
- **ツールの 3 つへの集約**: 旧来の多ツール群（`search_files` / `apply_patch` / `replace_selection` など、一時「実装済み」だったもの）は撤去。小型モデルのツール選択ミスを減らすため、`run_powershell` + 構造化 `write_file` / `edit_file` に統合した。検索・読み取りは PowerShell 経由で表現する。
- **実機精度ハーネス**: `AgentCapabilityHarness`（25 タスク × 多試行）で実モデルの成功率を計測し、決定論的な改善（パーサ補修・反復ガード・プロンプト汎用化）を回す体制を確立。
- **usage / トレース**: `ai.usage`（トークン数 + ロード/prefill/decode 自己計測）をトレースに記録し、AI バーへライブ表示。Trace セッションで閲覧可能。
- **UI 基盤の拡充**: ワークスペース切替、ペイン分割/リサイズ/並べ替え、マルチモニタ跨ぎ最大化、Diff セッション（side-by-side 表示）、Markdown プレビュー（mermaid 対応）、テーマ切替。
- **Git パネルの刷新**: 軽量ポーリングによるリアルタイム更新、変更行クリックでファイル/差分を開く、複数選択での一括操作。
- **ステージ（舞台/袖）モード**: ライブミニチュア表示、配役モード（主役＋サブ最大 2）、袖の活動バッジ（OSC133 ShellCommandActivity 連動、設計書 §24）、レイアウト復元。
- **コマンド系 UX**: コマンドコンポーザ（ターミナル下部組込）＋ペグボード（サイドバー）、コマンドパレット、キーボードショートカットの再割り当て、設定の中央オーバーレイ化、`/model` でのモデル切替。
- **EditorSupport プロバイダ**: CSV/TSV（自作 VGrid）・画像などファイル種別ごとの専用ビュー、タブのピン留め。
- **起動高速化**: 初フレーム 2038ms → 1077ms（プロファイラ + 遅延読込）。

## Priority 1: エージェントの精度と速度（ローカル LLM の核心）

### ハーネス駆動の精度改善（継続）

- 失敗モードの分類 → 決定論的な対策（プロンプトではなくパーサ・ガード・ツール側の修正）を優先。
- タスクセットの拡充と、評価過適合の監視（プロンプト変更が特定タスクだけに効いていないか）。

精度の物差しがあることが最大の資産。改善はすべてハーネスの数字で判断する。

### コンテキスト管理の強化

- 現状はトリムのみ。**要約・コンパクション**を導入し、長い作業でも文脈を失わないようにする。
- ツール出力（特に `run_powershell` の長い stdout）の賢い切り詰め・要約。

小型モデルはコンテキスト窓が実質的なボトルネック（8192 で運用中）。トリムで消える情報が失敗の一因になる。

### 推論の高速化・モデル選択肢の拡大

- GPU / NPU 実行（DirectML / QNN など ORT-GenAI の他 EP）の検証。CPU では 8B クラスが非実用（〜262s/ターン）と実測済みで、ハードを使えれば品質の上限が上がる。
- モデルカタログの拡充（ORT-GenAI 互換 = `genai_config.json` を持つビルドのみ）。

## Priority 2: ツールの慎重な拡張

ツールを増やすほど小型モデルの選択ミスが増えるため、**追加は 1 つずつハーネスで検証**してから。

### ブラウザ操作ツール

- `IBrowserService` のアダプタは実装済みだが、エージェントへは未接続。
- `open_url` / `get_page_text` 程度の最小セットから。Web アクセスは明示承認を必須に。

### UI 操作（フォーカス・タブ）

- AI がユーザーへ見せたいペインの前面化・該当ファイルのタブ表示。
- まずはツール化せず、ツール実行の副作用（`write_file` がエディタで開く等）として広げる方が小型モデルには安全。

## Priority 3: 安全性と運用性

### 権限プロファイル

- Read-only / Workspace-write / Command-limited / Full-manual のモード別制限。
- 現状は承認カード + 正規表現ブロックリスト + ファイルツールのワークスペース内制限。`run_powershell` 経由の書き込み（`Set-Content` 等）はブロックリストでしか守れていないため、モードで明確に縛りたい。

### コマンド実行のサンドボックス強化

- ワークスペース外への書き込み検知（pwsh 経由を含む）。
- ネットワーク使用の明示承認。
- 長時間実行コマンドのタイムアウトと停止 UI。
- 環境変数や秘密情報をログ・トレースへ残さないフィルタ。

### トレースのプライバシー管理

- 保持日数・最大サイズの設定、セッション単位の削除。
- ツール出力やファイル内容のマスク、エクスポート前の秘匿情報スキャン。

トレースはプロンプト・ファイル内容・コマンド出力を含むため、閲覧機能（実装済み）の次は管理機能が必要。

## Priority 4: 記憶と改善サイクル

### 長期記憶 / プロジェクト知識ベース

- ユーザー方針、プロジェクト固有ルール、過去の解決策を保存。
- embeddings + SQLite などでローカル検索し、セッション開始時に関連知識を注入。
- ローカル LLM の強み（プライベート知識を外に出さない）を活かせる領域。

### プレイバック

- トレースを使った読み取り専用の再生（ツール再実行なし）。
- ステップ実行、速度変更、タイムライン表示。

## Priority 5: 拡張性

### MCP / プラグイン式ツール拡張

- MCP クライアントとして外部ツールを接続。
- ツールごとの権限・承認要否を設定 UI で管理。
- ただし Priority 2 と同じ理由で、小型モデルではツール数自体がコストになる点に注意。動的なツール絞り込み（タスクに応じた提示）とセットで検討する。

### マルチエージェント

- Planner / Coder / Reviewer などの役割分担、`agentId` 付きトレース。
- 単一エージェントの精度（Priority 1）が安定してから段階導入。

## Suggested Next Steps

1. コンテキストのコンパクション（要約）を設計し、ハーネスの長尺タスクで効果を計測する。
2. 権限プロファイルの設定モデルを追加し、既存の承認 UI と安全ポリシーに接続する。
3. DirectML（GPU）実行を試し、Qwen3-4B 以上のクラスが実用速度に乗るか検証する。
4. ブラウザ最小ツール（`open_url` / `get_page_text`）を追加し、ハーネスで選択ミス率の悪化がないか確認する。
